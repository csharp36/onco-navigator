## 1. System Classification

**System Type:** Hybrid (Structured Extraction + Content Generation)

**Description:**
This system uses Claude AI for two distinct purposes within a HIPAA-compliant oncology care pathway monitoring application: (1) Document processing — classifying clinical PDFs (pathology reports, radiology reports, referral letters, operative notes, lab results) and extracting structured fields (document type, MRN, patient name, DOB, event type, event date) from unstructured clinical text. (2) Alert enhancement — generating plain-language deviation descriptions and suggested corrective actions for non-standard pathway deviations that don't match pre-defined template text. Users are nurse navigators and care coordinators at an oncology practice. "Good" means: correct document classification, accurate field extraction, clinically appropriate alert language, and graceful degradation when AI is unavailable.

**Critical Failure Modes:**
1. **Misclassification of document type** — A pathology report classified as a radiology report leads to wrong care event type pre-fill, potentially recording incorrect pathway progress
2. **Incorrect patient matching** — Extracting wrong MRN or patient name from a document could attach clinical data to the wrong patient — a patient safety issue
3. **PHI leakage in alert generation** — Alert generation prompts must contain zero PHI; any patient-identifiable information in prompts violates the Phase 2 zero-PHI design boundary
4. **Silent failure without fallback** — If Claude API fails and the circuit breaker doesn't engage, users see errors instead of the manual classification dropdown or template alert text
5. **Clinically misleading alert text** — Claude-generated alert descriptions that misrepresent the deviation could lead nurses to take incorrect corrective actions

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Healthcare -- Medical Oncology (Breast, Lung, Colorectal Cancer Care Coordination)

**User Population:** Nurse navigators (ONN-CG certified or equivalent), care coordinators, and medical records staff at an oncology practice. These users are clinically trained professionals who currently process incoming clinical documents manually, matching them to patients and recording care events by hand. They are not AI/ML practitioners -- they need the system to be transparent, predictable, and fast to override when wrong.

**Stakes Level:** High

**Output Consequence:** When the AI classifies a document and pre-fills a care event form, a nurse or care coordinator reviews and confirms before saving. An incorrect classification or patient match that slips past human review could record the wrong care event against a patient's pathway, causing the monitoring system to either (a) suppress a legitimate alert because it thinks a step was completed, or (b) generate a false alert for a step that was not actually done. Both outcomes undermine the core value proposition: preventing patients from falling through the cracks. For alert text generation, misleading deviation descriptions could direct a nurse's attention to the wrong corrective action, delaying appropriate follow-up for a cancer patient whose treatment timeline is clinically significant.

### What Domain Experts Evaluate Against

**Dimension: Document type accuracy**
Good (domain expert would accept): A pathology report from an outside lab is classified as PATHOLOGY_REPORT regardless of whether it is titled "Surgical Pathology Report," "Pathology Consultation," "Cytology Report," or uses institution-specific naming conventions. The classification matches what a medical records clerk would file it as.
Bad (domain expert would flag): A radiology report that includes pathology findings in the impression section is classified as PATHOLOGY_REPORT instead of IMAGING. A referral letter that mentions lab values is classified as LAB_WORK.
Stakes: Critical
Source: Medical records classification practice -- misfiling a clinical document leads to incorrect pathway step recording and downstream alert suppression or false alerting.

**Dimension: Patient identifier extraction fidelity**
Good (domain expert would accept): MRN is extracted exactly as printed on the document, including leading zeros and formatting. Patient name matches document header even when the document uses maiden name, hyphenated name, or "Last, First Middle" format. DOB is extracted in the correct date format without transposition errors (e.g., not swapping month/day).
Bad (domain expert would flag): MRN "00847293" extracted as "847293" (dropped leading zeros). Name "Garcia-Lopez, Maria" extracted as "Garcia, Maria Lopez." DOB "03/11/1962" extracted as "11/03/1962" (month/day swap). Any of these errors break the multi-field patient matching logic.
Stakes: Critical
Source: Patient identification is a Joint Commission National Patient Safety Goal (NPSG.01.01.01). Wrong-patient document attachment is a sentinel event category.

**Dimension: Care event field pre-fill correctness**
Good (domain expert would accept): An operative note for a lumpectomy pre-fills event type as SURGERY, event date as the date of the procedure (not the report date or dictation date), and status as COMPLETED. The nurse sees the pre-filled form and confirms without correction.
Bad (domain expert would flag): Event date populated with the report dictation date instead of the actual procedure date. Event type inferred from a keyword in the body text rather than the document's clinical context (e.g., a radiology report mentioning "post-surgical changes" pre-fills as SURGERY).
Stakes: High
Source: Oncology care pathway timing depends on actual event dates. A surgery recorded 10 days late shifts all downstream time windows, generating false delay alerts for subsequent steps.

**Dimension: Alert language clinical appropriateness**
Good (domain expert would accept): "Radiation therapy planning has not been scheduled within the expected 4-6 weeks following completed lumpectomy. This may indicate a referral gap to radiation oncology." The description names the specific pathway steps involved, states the time window, and suggests a concrete follow-up without prescribing clinical action.
Bad (domain expert would flag): "Patient is overdue for radiation treatment" (too vague, no actionable context). "Patient requires immediate radiation therapy" (prescribes clinical action -- violates human-in-the-loop boundary). "Treatment delay detected" (generic, does not help the nurse understand what to do).
Stakes: High
Source: Oncology Navigation Standards of Professional Practice (AONN+, 2022) -- navigators need specific, actionable information to coordinate across facilities and providers. Vague alerts waste time; prescriptive alerts overstep the system's role.

**Dimension: OCR extraction completeness on degraded documents**
Good (domain expert would accept): A faxed pathology report with moderate scan artifacts still yields the document type, patient name block, and MRN. The system signals low confidence and presents extracted fields for nurse verification rather than silently accepting partial data.
Bad (domain expert would flag): System extracts partial text from a poor-quality fax, fills in only the document type, and presents a "Ready" status without indicating that MRN and patient name could not be extracted. Nurse assumes matching failed when it was actually an extraction failure.
Stakes: Medium
Source: Published research on OCR accuracy in cancer pathology report extraction shows that character recognition errors on scanned documents directly impact downstream classification accuracy, particularly for numeric identifiers like MRN and date fields.

### Known Failure Modes in This Domain

1. **Institutional format variation in clinical documents.** Pathology reports, radiology reports, and operative notes have no universal format. Each external facility (hospital, imaging center, reference lab) uses different templates, headings, section ordering, and terminology. A model trained or prompted against one institution's format may consistently misclassify or mis-extract from another institution's documents. At a community oncology practice receiving documents from 10-20 external facilities, this variation is the norm, not the exception.

2. **OCR degradation on multiply-faxed documents.** Clinical documents frequently arrive as faxes of faxes -- a referring hospital faxes a report to the oncology practice, which may itself have been faxed from the originating lab. Each fax generation degrades resolution, introduces scan artifacts (lines, speckle, skew), and reduces OCR accuracy. Research on cancer pathology report extraction specifically found that OCR errors on fields like primary site and histological type are not negligible and compound downstream. The Tesseract-first, Claude-vision-fallback pipeline (D-02) directly addresses this, but the confidence threshold for escalation is a critical tuning parameter.

3. **Date field ambiguity in clinical documents.** Clinical documents contain multiple dates -- procedure date, report date, dictation date, transcription date, received date. A pathology report for a biopsy performed on March 5 may be dictated March 7, transcribed March 10, and faxed to the oncology practice March 14. The correct date for the care event is the procedure date (March 5), but it may appear in the body text while the header shows the report date. Extracting the wrong date shifts all downstream pathway time windows.

4. **Context window and longitudinal blindness in alert generation.** When generating alert text for non-standard deviations, Claude receives a snapshot of the current deviation context (anonymized). It has no longitudinal patient history. Published clinical LLM failure analysis documents a case where an LLM failed to account for longitudinal context because its memory window was too short. For Onco-Navigator, this means Claude may generate alert text that sounds reasonable in isolation but misses relevant context (e.g., suggesting a referral to radiation oncology when the patient already had a documented radiation oncology consult that was captured as a different event type). The human-in-the-loop design mitigates this, but the alert text should not be confidently wrong.

### Regulatory / Compliance Context

**HIPAA -- Business Associate Agreement with Anthropic (Hard Prerequisite)**
Document classification sends full extracted text (containing PHI) to the Claude API. Under HIPAA, Anthropic is acting as a Business Associate when processing ePHI. A signed BAA is required before any real patient documents are processed. Anthropic offers BAAs for Claude Enterprise and qualifying API customers via their Data Processing Addendum. The BAA covers the Messages API. It does not cover Batch API, Files API, beta features, or non-enterprise plans. The project must use the covered Messages API path. Alert generation maintains the zero-PHI boundary (D-14) and does not require BAA coverage for that specific function.

**HIPAA -- Audit Trail for AI-Processed Documents**
Every document classification and patient match must be logged with staff identity, timestamp, AI confidence indicators, and whether the nurse accepted or modified the AI's suggestion. This extends the existing Hibernate Envers audit trail (SEC-05). The audit must capture the pre-AI-edit and post-nurse-edit states to demonstrate human oversight.

**FDA -- Clinical Decision Support Exemption Analysis**
Under the 21st Century Cures Act and the FDA's March 2026 Final Guidance on Clinical Decision Support Software, CDS software is exempt from FDA device regulation if it meets four criteria: (1) not intended to acquire/process/analyze medical images or IVD signals, (2) intended to display, analyze, or print medical information, (3) intended to support (not replace) HCP clinical judgment, and (4) enables the HCP to independently review the basis for recommendations. Onco-Navigator's document classification and alert text generation likely qualify for this exemption because: the system processes document text, not diagnostic medical images; it displays extracted information for nurse review; it explicitly does not replace clinical judgment (human-in-the-loop is non-negotiable); and the nurse sees the source document alongside the AI's classification, enabling independent review. However, this is a regulatory analysis that should be reviewed by legal counsel familiar with FDA SaMD guidance before making claims about exemption status.

**No State-Specific AI Healthcare Regulations Identified**
Several US states are developing AI-in-healthcare regulations. None identified as directly applicable to a document classification and alert text tool at a single-practice pilot scale. This should be re-evaluated if the system scales to multi-practice deployment.

## 2. Framework Decision

**Selected Framework:** Spring AI 1.1.0 (`spring-ai-anthropic-spring-boot-starter`)

**Version:** 1.1.0 (GA, released November 2025)

**Rationale:**
Spring AI is the natural choice for this Java 21 + Spring Boot 3.5 project. It provides a `ChatClient` abstraction that wraps the official Anthropic Java SDK, supports structured output binding (JSON schema enforcement), and integrates with Spring Boot auto-configuration. The project already uses Spring Boot for all backend services, Spring Security for auth, and Spring Data JPA for persistence. Spring AI follows the same patterns (starters, properties, auto-config). If the model needs to change (e.g., Claude -> GPT-4 fallback), only the starter dependency and properties change — the ChatClient abstraction stays the same.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Direct `anthropic-java` SDK | No abstraction layer; model switching requires code changes; Spring AI wraps this SDK internally anyway |
| LangChain4j | Additional framework dependency with its own opinions; Spring AI is the Spring ecosystem's native choice |
| Python-based frameworks (LangChain, LlamaIndex, etc.) | Project is Java/Spring Boot — adding a Python service would mean a separate container, IPC overhead, and operational complexity |

**Vendor Lock-In Accepted:** Partial — Spring AI's `ChatClient` abstraction allows model switching by changing the starter (e.g., `spring-ai-openai-spring-boot-starter`). Anthropic-specific features (vision API for OCR fallback) would need adapter code.

### 3.3 Entry Point Pattern

The following creates a working `ChatClient` bean that can classify clinical documents. This is copy-paste ready once the Maven dependency and `application.yml` config are in place.

```java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.anthropic.AnthropicChatOptions;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class AiClientConfig {

    /**
     * ChatClient for document classification (low temperature, structured output).
     * Spring Boot auto-configures AnthropicChatModel from application.yml properties.
     * The ChatClient.Builder is auto-configured and injectable.
     */
    @Bean
    ChatClient documentClassificationClient(ChatClient.Builder builder) {
        return builder
                .defaultSystem("""
                    You are a clinical document classifier for an oncology practice.
                    You will receive the extracted text of a clinical document.
                    Classify the document type and extract patient identifiers \
                    and care event details.
                    Respond ONLY with the requested JSON structure. \
                    Do not include explanation.
                    """)
                .defaultOptions(AnthropicChatOptions.builder()
                        .temperature(0.1)
                        .maxTokens(1024)
                        .build())
                .build();
    }
}
```

**Minimal service call:**
```java
@Service
public class DocumentClassificationService {

    private final ChatClient classificationClient;

    public DocumentClassificationService(ChatClient classificationClient) {
        this.classificationClient = classificationClient;
    }

    public DocumentClassification classify(String extractedText) {
        return classificationClient.prompt()
                .user(extractedText)
                .call()
                .entity(DocumentClassification.class);
    }
}
```

### 3.4 Key Abstractions

| Abstraction | What It Does | How We Use It |
|-------------|-------------|---------------|
| `ChatClient` | High-level fluent API for prompt building, calling, and response parsing. Wraps the model and handles structured output, streaming, and advisors. | Primary interface for both document classification and alert generation calls. One `ChatClient` bean per use case (different system prompts, different temperature). |
| `ChatClient.Builder` | Auto-configured by Spring Boot. Creates `ChatClient` instances with default system prompts, options, and advisors. | Injected into `@Configuration` classes to build use-case-specific `ChatClient` beans. |
| `BeanOutputConverter` / `.entity()` | Converts LLM text output to Java records/classes. Generates JSON Schema from the record definition, appends format instructions to the prompt, and deserializes the response. | Document classification: LLM output deserialized into `DocumentClassification` record. Not used for alert generation (free text output). |
| `AnthropicChatOptions` | Runtime override of model, temperature, max tokens, cache options, and thinking mode. Can be set per-call without changing the bean default. | Per-call overrides when the same `ChatClient` needs different parameters (e.g., vision call with higher max tokens). |
| `Media` | Wraps binary content (images, PDFs) with MIME type for multimodal prompts. Accepted as part of `UserMessage`. | Claude vision OCR fallback: send scanned document image bytes as `Media` with `IMAGE_PNG` or `IMAGE_JPEG` MIME type. |

### 3.5 Pitfalls

1. **Default `max-tokens` is 500 -- too low for clinical extraction.** Spring AI's Anthropic integration defaults `max-tokens` to 500. A document classification response with 6+ fields in JSON regularly exceeds this, producing truncated JSON that fails to parse. Always set `max-tokens` explicitly in `application.yml` or via `AnthropicChatOptions`. For structured extraction: 1024. For alert generation: 2048.

2. **`spring.http.client.factory=jdk` required on Spring Boot 3.4.x.** A known Spring Boot bug causes HTTP client factory conflicts that break Spring AI calls silently. This property must be set in `application.properties`. Spring Boot 3.5 may resolve this -- verify during integration testing and remove if the issue does not reproduce. If omitted on an affected version, HTTP calls to Anthropic fail with opaque client errors.

3. **`BeanOutputConverter` is best-effort -- the model can still return malformed JSON.** The converter appends JSON Schema instructions to the prompt and deserializes the response, but the LLM is not token-level constrained to valid JSON. Anthropic supports native structured output mode on Claude 3.5+ when enabled via `AdvisorParams.ENABLE_NATIVE_STRUCTURED_OUTPUT`, which significantly improves reliability. Even with native mode, implement validation and retry logic -- partial network interruptions or extreme edge cases can still produce malformed output.

4. **Streaming + multimodal (images) causes premature connection close.** Known issue (GitHub #5011): pure text streaming works, but once an image is included in the message, a "connection prematurely closed" error occurs during streaming. For the Claude vision OCR fallback, always use `.call()` (synchronous), never `.stream()`. This is acceptable for our use case since OCR needs the full structured response before proceeding.

5. **Thinking mode requires `temperature: 1.0` and budget >= 1024 tokens.** If extended thinking is enabled (Claude 3.7+), temperature must be set to exactly 1.0 and the thinking token budget must be at least 1024. Violating either constraint causes an API error, not a graceful fallback. This project should NOT use thinking mode for document classification -- it adds 2-5 seconds of latency for a task that does not benefit from chain-of-thought reasoning.

6. **Prompt caching invalidation hierarchy.** Anthropic caches in order: system message, then tools, then conversation history. Changing anything in an earlier layer invalidates all later layers. This project does not use tool calling, but if tools are added later, be aware that modifying tool definitions invalidates the cached system prompt. Cache entries also have a minimum token threshold -- Haiku requires 4,096 tokens; Sonnet requires fewer (exact threshold not documented). Short system prompts may not be cacheable.

## 4. Implementation Guidance

### 4.1 Model Configuration

**Document Classification (Structured Extraction):**

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Model | `claude-sonnet-4-20250514` | Best cost/quality ratio for structured extraction. Supports native structured output. Significantly cheaper than Opus. |
| Temperature | `0.1` | Near-deterministic output for classification. The same document should be classified the same way every time. |
| Max Tokens | `1024` | JSON response for 8 fields fits well within 500 tokens, but buffer for edge cases (verbose field values, unusual document structures). |
| Top-P | not set (use default) | Temperature alone provides sufficient control for classification. |

**Alert Description Generation (Content Generation):**

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Model | `claude-sonnet-4-20250514` | Same model for operational simplicity. Alert descriptions are 2-4 sentences -- does not need Opus quality. |
| Temperature | `0.3` | Slightly creative for natural language, but constrained enough to avoid clinically misleading phrasing. |
| Max Tokens | `2048` | Alert description (2-4 sentences) plus suggested corrective action (1-3 bullet points). Buffer for thorough explanations of complex deviations. |

**Claude Vision OCR Fallback:**

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Model | `claude-sonnet-4-20250514` | Same model. Claude Sonnet handles vision extraction well. |
| Temperature | `0.1` | OCR-equivalent extraction must be deterministic. |
| Max Tokens | `4096` | Scanned documents can produce more text than digital PDFs. Generous buffer for full-page scans. |

### 4.2 Core Pattern: Document Classification

Single Claude call that classifies the document AND extracts patient identifiers and care event fields. Matches decision D-13 (full extracted text sent to Claude, BAA-covered).

**Output record (structured extraction target):**
```java
import com.fasterxml.jackson.annotation.JsonProperty;
import com.fasterxml.jackson.annotation.JsonPropertyOrder;

@JsonPropertyOrder({"documentType", "confidence", "mrn", "patientName",
                     "dateOfBirth", "eventType", "eventDate", "extractedNotes"})
public record DocumentClassification(

    @JsonProperty(required = true, value = "documentType")
    String documentType,  // PATHOLOGY_REPORT, RADIOLOGY_REPORT, LAB_RESULT,
                          // OPERATIVE_NOTE, REFERRAL_LETTER, UNKNOWN

    @JsonProperty(required = true, value = "confidence")
    String confidence,    // HIGH, MEDIUM, LOW

    @JsonProperty(value = "mrn")
    String mrn,           // nullable -- not all documents contain MRN

    @JsonProperty(value = "patientName")
    String patientName,   // nullable -- extracted as-is from document
```

**Classification service:**
```java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.stereotype.Service;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class DocumentClassificationService {

    private static final Logger log = LoggerFactory.getLogger(
            DocumentClassificationService.class);
    private static final int MAX_INPUT_TOKENS = 150_000;
    private static final int CHARS_PER_TOKEN_ESTIMATE = 4;

    private final ChatClient classificationClient;

    public DocumentClassificationService(ChatClient documentClassificationClient) {
        this.classificationClient = documentClassificationClient;
    }

    /**
     * Classify a clinical document and extract structured fields.
     * Called after text extraction (PDFBox or Tesseract).
     *
     * @param extractedText raw text from the document
     * @return structured classification, or null on failure (caller shows manual dropdown)
     */
    public DocumentClassification classify(String extractedText) {
        String truncated = truncateToTokenBudget(extractedText);

        try {
            return classificationClient.prompt()
                    .user(u -> u.text("""
                        Classify the following clinical document and extract \
                        patient identifiers.

                        DOCUMENT TEXT:
                        ---
                        {documentText}
                        ---

                        Extract the following fields:
                        - documentType: one of PATHOLOGY_REPORT, RADIOLOGY_REPORT, \
                          LAB_RESULT, OPERATIVE_NOTE, REFERRAL_LETTER, or UNKNOWN
                        - confidence: HIGH, MEDIUM, or LOW based on how clearly \
                          the document matches the type
                        - mrn: Medical Record Number if present, null otherwise
                        - patientName: full name if present, null otherwise
                        - dateOfBirth: patient DOB in yyyy-MM-dd format if present, \
                          null otherwise
                        - eventType: care event type (PATHOLOGY_REPORT, IMAGING, \
                          LAB_WORK, SURGERY, CONSULTATION) if determinable, null otherwise
                        - eventDate: date of the clinical event (NOT the report date \
                          or dictation date) in yyyy-MM-dd format if present, null otherwise
                        - extractedNotes: 1-2 sentence summary of key clinical findings
                        """)
                            .param("documentText", truncated))
                    .call()
                    .entity(DocumentClassification.class);

        } catch (Exception e) {
            // Log error without PHI -- do NOT log extractedText
            log.error("Document classification failed: {}", e.getMessage());
            return null;
        }
    }

    /**
     * Conservative character-based truncation.
     * ~4 chars per token for English clinical text.
     */
    private String truncateToTokenBudget(String text) {
        int maxChars = MAX_INPUT_TOKENS * CHARS_PER_TOKEN_ESTIMATE;
        if (text.length() <= maxChars) {
            return text;
        }
        log.warn("Document text truncated from {} to {} chars for classification",
                text.length(), maxChars);
        return text.substring(0, maxChars);
    }
}
```

```java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.stereotype.Service;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import java.util.List;

@Service
public class AlertGenerationAiService {

    private static final Logger log = LoggerFactory.getLogger(
            AlertGenerationAiService.class);

    private final ChatClient alertClient;

    public AlertGenerationAiService(ChatClient alertGenerationClient) {
        this.alertClient = alertGenerationClient;
    }

    /**
     * Generate deviation description and suggested action for a non-standard alert.
     * ZERO-PHI: Only anonymized clinical context is sent. No patient identifiers.
     *
     * @return generated text, or null on failure (caller uses template text)
     */
    public AlertText generateAlertDescription(
            String cancerType,
            String pathwayStepName,
            String deviationType,
            String timeWindowDays,
            List<String> completedSteps,
            List<String> missingSteps) {

        try {
            String response = alertClient.prompt()
                    .user(u -> u.text("""
                        A care pathway deviation has been detected.

                        Cancer type: {cancerType}
                        Current pathway step: {pathwayStepName}
                        Deviation type: {deviationType}
                        Expected time window: {timeWindowDays}
                        Completed steps: {completedSteps}
                        Missing/pending steps: {missingSteps}

                        Generate:
                        1. DESCRIPTION: A 2-4 sentence plain-language description of what \
                           deviated and why it matters clinically.
                        2. SUGGESTED_ACTION: 1-3 specific coordination actions the nurse \
                           navigator should take.

                        Format your response exactly as:
                        DESCRIPTION: [your description]
                        SUGGESTED_ACTION: [your suggested actions]
                        """)
                            .param("cancerType", cancerType)
                            .param("pathwayStepName", pathwayStepName)
                            .param("deviationType", deviationType)
                            .param("timeWindowDays", timeWindowDays)
                            .param("completedSteps", String.join(", ", completedSteps))
                            .param("missingSteps", String.join(", ", missingSteps)))
                    .call()
                    .content();

            return parseAlertResponse(response);

        } catch (Exception e) {
            log.error("Alert description generation failed: {}", e.getMessage());
            return null;
        }
    }

    private AlertText parseAlertResponse(String response) {
        if (response == null
                || !response.contains("DESCRIPTION:")
                || !response.contains("SUGGESTED_ACTION:")) {
            return null;
        }

        int descStart = response.indexOf("DESCRIPTION:") + "DESCRIPTION:".length();
        int actionStart = response.indexOf("SUGGESTED_ACTION:");
        String description = response.substring(descStart, actionStart).trim();
        String suggestedAction = response.substring(
                actionStart + "SUGGESTED_ACTION:".length()).trim();

        if (description.isEmpty()) {
            return null;
        }
        return new AlertText(description, suggestedAction);
    }

    public record AlertText(String deviationDescription, String suggestedAction) {}
}
```

### 4.4 Core Pattern: Claude Vision OCR Fallback

When Tesseract OCR produces low-quality output (below confidence threshold), escalate to Claude vision for text extraction. Uses `Media` to send the document image directly.

```java
import org.springframework.ai.anthropic.AnthropicChatModel;
import org.springframework.ai.anthropic.AnthropicChatOptions;
import org.springframework.ai.chat.messages.UserMessage;
import org.springframework.ai.chat.prompt.Prompt;
import org.springframework.ai.model.Media;
import org.springframework.core.io.ByteArrayResource;
import org.springframework.stereotype.Service;
import org.springframework.util.MimeType;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import java.util.List;

@Service
public class ClaudeVisionService {

    private static final Logger log = LoggerFactory.getLogger(ClaudeVisionService.class);

    private final AnthropicChatModel chatModel;

    public ClaudeVisionService(AnthropicChatModel chatModel) {
        this.chatModel = chatModel;
    }

    /**
     * Extract text from a scanned document image using Claude vision.
     * Uses synchronous .call() -- streaming + images has known issues (GitHub #5011).
     *
     * @param imageBytes raw image bytes (JPEG or PNG)
     * @param mimeType   image MIME type (MimeTypeUtils.IMAGE_PNG or IMAGE_JPEG)
     * @return extracted text, or empty string on failure
     */
    public String extractTextFromImage(byte[] imageBytes, MimeType mimeType) {
        var imageResource = new ByteArrayResource(imageBytes);

        var userMessage = new UserMessage(
                """
                Extract ALL text from this scanned clinical document image.
                Preserve the original structure as much as possible.
                Include headers, patient information fields, dates, and \
                clinical content.
                If any text is illegible, indicate it with [illegible].
                Return only the extracted text, no commentary.
                """,
                List.of(new Media(mimeType, imageResource))
        );

        try {
            var response = chatModel.call(new Prompt(
                    List.of(userMessage),
                    AnthropicChatOptions.builder()
                            .model("claude-sonnet-4-20250514")
                            .temperature(0.1)
                            .maxTokens(4096)
                            .build()
            ));
            return response.getResult().getOutput().getText();
        } catch (Exception e) {
            log.error("Claude vision extraction failed: {}", e.getMessage());
            return "";  // Caller opens blank care event form (D-04)
        }
    }
}
```

### 4.6 State Management

**Stateless per-request.** Each document classification and each alert generation call is independent. No conversation history, no memory, no session state.

- Document classification: one call per uploaded document. Input is extracted text, output is structured JSON.
- Alert generation: one call per non-standard deviation. Input is anonymized context, output is description text.
- Vision OCR: one call per failed Tesseract extraction. Input is image bytes, output is extracted text.

No `MessageChatMemoryAdvisor` or similar memory advisors should be configured.

### 4.7 Context Window Strategy

Claude Sonnet 4 has a 200K token context window. Clinical documents vary in size:

| Document Type | Typical Size | Token Estimate |
|---------------|-------------|----------------|
| Lab result (1-2 pages) | 500-2,000 chars | 125-500 tokens |
| Radiology report (1-3 pages) | 1,000-5,000 chars | 250-1,250 tokens |
| Pathology report (2-5 pages) | 2,000-15,000 chars | 500-3,750 tokens |
| Operative note (3-10 pages) | 5,000-40,000 chars | 1,250-10,000 tokens |
| Multi-page scanned fax | 10,000-80,000 chars | 2,500-20,000 tokens |

**Strategy:**
- Most clinical documents fit easily within the context window, even with system prompt and format instructions.
- The `truncateToTokenBudget()` method provides a safety net for unusually large documents (e.g., a 50-page operative note with full surgical history).
- Truncation threshold: 150,000 tokens (~600,000 chars) leaves room for system prompt (~500 tokens), format instructions (~300 tokens), and response tokens (1,024).
- Truncation is from the end. Clinical documents front-load classification-relevant information (document type header, patient demographics, report date).
- If a document is truncated, a warning is logged with the document ID (not content) for monitoring.

**Alert generation context is small by design.** The zero-PHI boundary (D-14) means only anonymized metadata is sent: cancer type, step names, deviation type, time window. This never approaches context limits (~200 tokens per call).

**Retry logic for output validation failures:**

Spring AI's built-in retry (`spring.ai.retry.*`) handles transient HTTP failures (5xx, timeouts). For output validation failures (malformed JSON, missing required fields), implement application-level retry:

```java
public DocumentClassification classifyWithRetry(String extractedText, int maxRetries) {
    for (int attempt = 1; attempt <= maxRetries; attempt++) {
        try {
            DocumentClassification result = classificationClient.prompt()
                    .user(extractedText)
                    .call()
                    .entity(DocumentClassification.class);

            // Validate business rules beyond JSON schema
            if (result.documentType() == null || result.documentType().isBlank()) {
                log.warn("Attempt {}/{}: documentType is blank", attempt, maxRetries);
                continue;
            }
            return result;

        } catch (Exception e) {
            log.warn("Attempt {}/{} failed: {}", attempt, maxRetries, e.getMessage());
            if (attempt == maxRetries) {
                log.error("All {} classification attempts exhausted", maxRetries);
                return null;
            }
        }
    }
    return null;
}
```

- **Retry count:** 2 retries (3 total attempts). Clinical document classification is not time-critical enough to skip retries, but 3 attempts keeps total latency under 15 seconds.
- **What to log:** Attempt number, failure reason (parse error vs. API error), document entity ID (NOT document text content -- that is PHI).
- **When to surface:** After all retries fail, return null. The caller (controller or Temporal activity) surfaces the manual classification dropdown to the user.

### 4b.2 Async-First Design

**How async works in Spring AI + Spring Boot 3.5 with Java 21:**

Spring Boot 3.5 with Java 21 supports virtual threads (`spring.threads.virtual.enabled=true`), which is already configured for this project's Temporal workers. Virtual threads make blocking `.call()` operations lightweight -- each Claude API call blocks a virtual thread, not a platform thread. The JVM scheduler yields the virtual thread during I/O wait (the HTTP call to Anthropic), freeing the carrier thread for other work.

For Spring AI specifically:
- **`.call()`** is synchronous/blocking. With virtual threads enabled, this blocks a virtual thread (effectively free in terms of platform thread consumption). This is the correct choice for document classification and alert generation where the full response is needed before proceeding.
- **`.stream()`** returns `Flux<ChatResponse>` (Project Reactor). Useful for streaming long responses to the frontend in real-time. Requires `spring-boot-starter-webflux` on the classpath.

**Recommended approach for this project:**

```java
// Document classification: synchronous .call() with virtual threads.
// The Temporal activity thread is already a virtual thread when
// spring.threads.virtual.enabled=true is set.
@ActivityImpl(workers = "ai-worker")
public class DocumentProcessingActivityImpl implements DocumentProcessingActivity {

    private final DocumentClassificationService classificationService;

    public DocumentProcessingActivityImpl(
            DocumentClassificationService classificationService) {
        this.classificationService = classificationService;
    }

    @Override
    public DocumentClassification classifyDocument(String extractedText) {
        // Blocks a virtual thread while waiting for Claude API response.
        // Virtual thread yields to carrier thread during network I/O.
        // No platform thread is consumed during the wait.
        return classificationService.classify(extractedText);
    }
}
```

**The one common mistake -- do NOT add unnecessary async wrappers:**

Do not wrap Claude calls in `CompletableFuture.supplyAsync()` or `@Async` when virtual threads are enabled. Virtual threads already provide non-blocking I/O at the JVM level. Adding `@Async` creates unnecessary thread pool management overhead, makes error propagation harder (exceptions wrapped in `CompletionException`), and makes Temporal activity retry/timeout semantics harder to reason about. The blocking `.call()` on a virtual thread is the correct and simplest approach.

**Stream vs. Await decision:**
- Use `.call()` (await) for document classification, alert generation, and vision OCR. All three need the complete response for validation, structured output parsing, or downstream processing.
- Use `.stream()` only if a future enhancement streams extraction progress to the user in real-time. Not in Phase 4 scope.

### 4b.3 Prompt Engineering Discipline

**System vs. user message separation:**

The system message defines the LLM's role and constraints (set once at `ChatClient` bean creation). The user message provides the specific input for each call. Spring AI enforces this separation through `ChatClient.Builder.defaultSystem()` and `.user()`.

```java
// CORRECT: System message at bean creation, user message per-call.
@Bean
ChatClient classificationClient(ChatClient.Builder builder) {
    return builder
            .defaultSystem(ClassificationPrompts.SYSTEM_PROMPT)
            .build();
}

// Usage:
client.prompt()
    .user(u -> u.text(ClassificationPrompts.USER_TEMPLATE)
                .param("documentText", text))
    .call()
    .entity(DocumentClassification.class);
```

```java
// WRONG: System instructions mixed into user message.
// Wastes tokens, makes prompt caching impossible, violates separation.
client.prompt()
    .user("You are a classifier. Classify this: " + text)
    .call()
    .content();
```

**Prompt templates as constants:**

Store prompts in a dedicated class, not inline in service methods. This makes them testable, reviewable, and version-controlled independently of the service logic.

```java
public final class ClassificationPrompts {
    private ClassificationPrompts() {}

    public static final String SYSTEM_PROMPT = """

        You will receive the extracted text of a clinical document.

        care event details.

        explanation or commentary.

        For dates, always use yyyy-MM-dd format.
        For MRN, extract the exact string as it appears in the document, \
        including leading zeros.
        """;

    public static final String USER_TEMPLATE = """
        Classify the following clinical document and extract fields.

        DOCUMENT TEXT:
        ---
        {documentText}
        ---
        """;
}
```

**Few-shot examples for classification -- inline in system prompt:**

Include 1-2 condensed examples in the system prompt to anchor the expected output format. The system prompt with examples is cached via Anthropic prompt caching (`SYSTEM_ONLY` strategy), so the token cost of few-shot examples is paid only on the first call within a cache window.

```java
public static final String SYSTEM_PROMPT = """

    ...

    EXAMPLE 1:
    Input: "SURGICAL PATHOLOGY REPORT ... Patient: Jane Doe ... MRN: 00847293 ..."

    Input: "CT Chest with Contrast ... Patient: John Smith ..."
    Output: {"documentType":"RADIOLOGY_REPORT","confidence":"HIGH",...}
    """;
```

**Set `max-tokens` explicitly -- never leave unbounded in production:**

Always set `max-tokens` per use case. An unbounded or default-500 `max-tokens` causes either truncated responses (classification JSON cut off mid-field, producing a parse error) or unnecessary cost (model generating far more text than needed for alert descriptions).

| Use Case | `max-tokens` | Why |
|----------|-------------|-----|
| Document classification | 1024 | 8-field JSON. Buffer for verbose notes. |
| Alert generation | 2048 | 2-4 sentence description + 1-3 action bullets. |
| Vision OCR | 4096 | Full-page text extraction from scans. |

### 4b.4 Context Window Management

**Document classification (potential for large context):**

Clinical documents are generally small (1-10 pages, 500-40,000 characters), but edge cases exist: multi-page operative notes, bundled lab panels, or scanned fax batches. Claude Sonnet 4's 200K token context window handles virtually all single clinical documents without truncation.

Safety net strategy:
1. **Measure first.** Log the character count of every document sent for classification. After the pilot accumulates data, set the truncation threshold based on the 99th percentile observed size.
2. **Truncate from the end.** Clinical documents front-load classification-relevant information (document header, patient demographics, report date). Truncating the end preserves the most useful content.
3. **Never silently truncate.** Log a warning with the document entity ID (not content) when truncation occurs. This surfaces in monitoring for human review.

```java
private static final int MAX_INPUT_TOKENS = 150_000;
private static final int CHARS_PER_TOKEN_ESTIMATE = 4;

private String truncateToTokenBudget(String text) {
    int maxChars = MAX_INPUT_TOKENS * CHARS_PER_TOKEN_ESTIMATE;
    if (text.length() <= maxChars) {
        return text;
    }
    log.warn("Document truncated from {} to {} chars", text.length(), maxChars);
    return text.substring(0, maxChars);
}
```

**Alert generation (always small context):**

The zero-PHI boundary ensures alert generation context is always small (~200 tokens: cancer type, step names, deviation type, time window). No truncation or chunking needed. This is a deliberate benefit of the anonymized design from D-14.

**Prompt caching for cost reduction:**

Enable Anthropic prompt caching on the system message. The document classification system prompt is identical across all calls. With `SYSTEM_ONLY` caching, repeated calls within the cache window pay only for the user message tokens.

#### E-01: Document Type Classification Accuracy

**Priority:** Critical
**Failure Mode Addressed:** FM-1 (Misclassification of document type)
**Requirement:** DOC-02

PASS: The system classifies a pathology report as PATHOLOGY_REPORT regardless of whether the source document is titled "Surgical Pathology Report," "Pathology Consultation," "Cytology Report," or uses an institution-specific heading. A radiology report containing incidental pathology findings in the impression section is classified as RADIOLOGY_REPORT (not PATHOLOGY_REPORT). Classification matches what a medical records clerk at the pilot practice would file it as.

FAIL: A radiology report that mentions pathology findings is classified as PATHOLOGY_REPORT. A referral letter that includes lab values is classified as LAB_RESULT. Any document whose classification does not match the ground-truth label in the reference dataset.

**Measurement:** Code -- exact string match of `documentType` field against ground-truth label in reference dataset. Compute accuracy (correct / total), precision, and recall per document type. Target: >=90% overall accuracy, >=85% per-type accuracy on the reference dataset.

```java
// JUnit 5 parameterized test pattern
```

#### E-02: Patient Identifier Extraction Fidelity

**Priority:** Critical
**Failure Mode Addressed:** FM-2 (Incorrect patient matching)
**Requirement:** DOC-03

PASS: MRN "00847293" is extracted as "00847293" with leading zeros preserved. Patient name "Garcia-Lopez, Maria" is extracted as-is from the document header. DOB "03/11/1962" is extracted as "1962-03-11" (correct ISO 8601 conversion, no month/day transposition). When a field is not present in the document, the extracted value is null (not guessed).

FAIL: MRN "00847293" is extracted as "847293" (leading zeros dropped). Name "Garcia-Lopez, Maria" is extracted as "Garcia, Maria Lopez" (hyphenation lost). DOB month and day are transposed. A field that does not exist in the document is populated with a fabricated value.

**Measurement:** Code -- exact string comparison for MRN (including leading zeros), normalized string comparison for patient name (trim whitespace, case-insensitive but structure-preserving), ISO 8601 date comparison for DOB. Target: 100% exact MRN match, >=95% name match, 100% DOB match on reference dataset.

```java
@ParameterizedTest
```

#### E-04: Alert Language Clinical Appropriateness

**Priority:** High
**Failure Mode Addressed:** FM-5 (Clinically misleading alert text)
**Requirement:** AI-02, AI-03

PASS: Alert text names the specific pathway steps involved (e.g., "lumpectomy" and "radiation therapy planning"), states the expected time window (e.g., "4-6 weeks"), and suggests a concrete coordination action (e.g., "Contact radiation oncology scheduling to verify referral status") without prescribing clinical treatment. Language is appropriate for an experienced oncology nurse navigator.

FAIL: Alert text is vague ("Treatment delay detected"). Alert text prescribes clinical action ("Patient requires immediate radiation therapy"). Alert text invents clinical details not provided in the context. Alert text is generic and does not help the nurse understand what to do.

**Measurement:** LLM Judge (calibrated against human) + Human review.

LLM judge rubric (5-point scale, applied via Promptfoo):
- **5 (Excellent):** Names specific pathway steps, states time window, suggests concrete coordination action, no prescriptive clinical language, no fabricated details.
- **4 (Good):** Names pathway steps and time window, suggestion is directionally correct but slightly generic.
- **3 (Acceptable):** Identifies the deviation correctly but suggestion lacks specificity. Nurse would need to interpret.
- **2 (Poor):** Vague description ("delay detected") OR suggestion is not actionable OR contains minor inaccuracy.
- **1 (Unacceptable):** Prescribes clinical treatment, fabricates clinical details, or is misleading about what deviated.

Target: >=4.0 average score across reference dataset. Zero scores of 1 (unacceptable).

Human calibration: Nurse navigator scores 10 reference examples. LLM judge scores the same 10. Require Pearson correlation >=0.7 before trusting LLM judge in CI.

#### E-05: OCR Extraction Completeness on Degraded Documents

**Priority:** Medium
**Failure Mode Addressed:** Known Domain Failure Mode #2 (OCR degradation on multiply-faxed documents)
**Requirement:** DOC-02 (classification must work on scanned documents)

PASS: A faxed pathology report with moderate scan artifacts still yields the document type, patient name, and MRN. When extraction is partial, the confidence field is LOW or MEDIUM (not HIGH), signaling to the nurse that manual verification is needed.

FAIL: System extracts partial text, fills in only document type, and returns confidence HIGH. Or system returns null for all fields without indicating that extraction failed (nurse cannot distinguish "no MRN in document" from "could not read the document").

**Measurement:** Code -- on the degraded-document subset of the reference dataset, measure: (a) field extraction rate (non-null fields / expected fields), (b) confidence field correctly reflects extraction quality (LOW or MEDIUM when fields are missing). Target: >=70% field extraction rate on degraded documents, 100% confidence accuracy (never HIGH when fields are missing).

#### E-06: PHI Boundary Enforcement in Alert Generation

**Priority:** Critical
**Failure Mode Addressed:** FM-3 (PHI leakage in alert generation)
**Requirement:** AI-02 zero-PHI boundary, SEC-06

PASS: Alert generation prompts contain only anonymized clinical context: cancer type, pathway step names, deviation type, time window, completed/missing step names. No patient name, MRN, DOB, or any other identifier appears in the prompt sent to Claude.

FAIL: Any patient-identifiable information (name, MRN, DOB, address, phone number) appears in the prompt text sent to the Claude API for alert generation.

**Measurement:** Code -- unit test that intercepts the prompt string before it is sent to Claude (mock the ChatClient) and asserts no PHI patterns are present. Regex check for MRN patterns (`\d{6,10}`), date-of-birth patterns, and name patterns from test fixtures. This is a hard pass/fail with zero tolerance.

```java
@Test
void alertGenerationPromptContainsNoPhi() {
    // Capture the prompt sent to Claude via mock
    ArgumentCaptor<Prompt> promptCaptor = ArgumentCaptor.forClass(Prompt.class);
    when(mockChatModel.call(promptCaptor.capture())).thenReturn(mockResponse);

    alertService.generateAlertDescription(
            "Breast Cancer Stage II",
            "Radiation Therapy Planning",
            "MISSING_EVENT",
            "28-42 days",
            List.of("Surgery", "Pathology Review"),
            List.of("Radiation Therapy Planning"));

    String promptText = promptCaptor.getValue().getContents();
    // Must NOT contain any PHI patterns
    assertThat(promptText).doesNotContainPattern("\\b\\d{3}-\\d{2}-\\d{4}\\b"); // SSN
    assertThat(promptText).doesNotContainPattern("\\bMRN[:\\s]*\\d+\\b");
    assertThat(promptText).doesNotContain("Jane Doe", "John Smith"); // test fixture names
    // Must only contain anonymized context
    assertThat(promptText).contains("Breast Cancer Stage II");
    assertThat(promptText).contains("Radiation Therapy Planning");
}
```

#### E-07: Circuit Breaker Fallback Behavior

**Priority:** Critical
**Failure Mode Addressed:** FM-4 (Silent failure without fallback)
**Requirement:** AI-04

PASS: When the Claude API is unavailable (connection timeout, 500 error, rate limit), the Resilience4j circuit breaker opens after the configured failure threshold. Document classification falls back to the manual classification dropdown -- the nurse sees the form with empty fields and the dropdown pre-selected to "Select document type." Alert generation falls back to template text -- the alert is created with the standard deviation template, not left blank.

FAIL: Claude API failure results in an error page, a spinner that never resolves, or a blank alert with no description. The circuit breaker does not engage and each subsequent request also fails.

**Measurement:** Code -- integration test that simulates Claude API failure (WireMock returning 500) and verifies: (a) circuit breaker transitions to OPEN after N failures, (b) document classification returns null (triggering manual dropdown in the controller), (c) alert generation returns null (triggering template text in AlertGenerationActivityImpl), (d) after circuit breaker timeout, system retries (HALF_OPEN state).

```java
@Test
void circuitBreakerOpensOnRepeatedFailure() {
    // WireMock returns 500 for Claude API
    stubFor(post("/v1/messages").willReturn(serverError()));

    // Exhaust failure threshold
    for (int i = 0; i < 3; i++) {
        DocumentClassification result = classificationService.classify(sampleText);
        assertThat(result).isNull();
    }

    // Circuit should now be OPEN -- next call should fail fast
    Instant start = Instant.now();
    DocumentClassification result = classificationService.classify(sampleText);
    Duration elapsed = Duration.between(start, Instant.now());

    assertThat(result).isNull();
    assertThat(elapsed).isLessThan(Duration.ofMillis(100)); // Fast fail, not timeout
}
```

### 5.5 Alert Language LLM Judge Calibration Protocol

Because E-04 (Alert Language Clinical Appropriateness) uses an LLM judge, it requires calibration before the judge can be trusted in CI.

**Calibration steps:**
1. Generate alert descriptions for all 10 alert test cases using the actual prompt and model configuration.
2. Have the nurse navigator score each output on the 5-point rubric defined in E-04.
3. Run the same outputs through the Promptfoo LLM judge rubric.
4. Compute Pearson correlation between human scores and LLM judge scores.
5. If correlation >= 0.7, the LLM judge is calibrated and can be used in CI.
6. If correlation < 0.7, adjust the rubric wording and repeat. Common fixes: more explicit examples of each score level, tighter definitions of "prescriptive" vs. "coordination" language.

**Recalibration trigger:** Any prompt change to the alert generation system prompt or user template requires re-running the calibration protocol.

## 6. Guardrails

> Guardrails are classified as Online (runs on every request, real-time) or Offline (sampled batch, feeds improvement loop). Each maps to a critical failure mode.

### 6.1 Online Guardrails

Online guardrails run on every AI call in production. They add latency, so each must justify its cost. All three below address catastrophic failure modes.

| ID | Guardrail | Failure Mode | Implementation | Latency Impact |
|----|-----------|-------------|----------------|----------------|
| G-01 | **Resilience4j Circuit Breaker** | FM-4: Silent failure without fallback | Wraps all Claude API calls. Closed -> Open after 3 failures in 30 seconds. Half-open after 60 seconds. Open state returns null immediately (fast-fail). Document classification fallback: manual dropdown. Alert generation fallback: template text. | <1ms (state check only) |
| G-02 | **Classification Confidence Gate** | FM-1: Misclassification, FM-2: Incorrect matching | If `DocumentClassification.confidence` is LOW, the pre-filled form displays a visual warning banner ("Low confidence -- please verify all fields") and the user must explicitly confirm each extracted field before saving. If confidence is MEDIUM, a subtle indicator appears but no blocking confirmation is required. | 0ms (post-response UI logic) |
| G-03 | **PHI Prompt Sanitization Check** | FM-3: PHI leakage in alert generation | Before sending the alert generation prompt to Claude, a code check validates that the prompt string does not match PHI patterns: MRN regex (`\d{6,10}` adjacent to "MRN"), date-of-birth patterns (adjacent to "DOB" or "birth"), and patient name patterns (from the source alert's patient record). If any match is found, the call is blocked and falls back to template text. This is defense-in-depth on top of the architectural zero-PHI boundary. | <5ms (regex scan of ~200 token string) |
| G-04 | **Structured Output Schema Validation** | FM-1: Misclassification (malformed output) | After `BeanOutputConverter` deserialization, validate that `documentType` is a member of the allowed enum set and `confidence` is one of HIGH/MEDIUM/LOW. If validation fails after 3 retry attempts, return null (manual dropdown fallback). | <1ms (enum check) |

### 6.2 Offline Flywheel

Offline flywheel processes run on sampled data, not every request. They feed the improvement loop: identifying quality issues, building the reference dataset, and tuning prompts.

| ID | Flywheel | What It Feeds | Cadence | Owner |
|----|----------|--------------|---------|-------|
| F-01 | **Nurse Override Rate Tracking** | Prompt improvement, reference dataset expansion | Continuous (logged per event) | Care Coordinator / Dev team |
| F-02 | **Alert Text Quality Sampling** | LLM judge recalibration, prompt tuning | Weekly (10 random AI-generated alerts) | Nurse Navigator |
| F-03 | **Classification Confusion Matrix** | Per-type accuracy trends, prompt few-shot example updates | Monthly (all classifications from prior month) | Dev team |
| F-04 | **Degraded Document Audit** | OCR threshold tuning, vision fallback trigger calibration | Monthly (all LOW confidence classifications) | Dev team + Care Coordinator |

**F-01: Nurse Override Rate Tracking (Primary Quality Signal)**

The most important quality signal in this system. Every time a nurse modifies an AI-extracted field before saving a care event, that constitutes an override. The audit trail (Hibernate Envers) already captures pre-AI and post-nurse states (Section 1b, HIPAA Audit Trail requirement).

Tracked overrides:
- `documentType` changed by nurse (classification was wrong)
- `mrn` changed by nurse (extraction error)
- `patientName` changed by nurse (extraction error)
- `eventType` changed by nurse (mapping error)
- `eventDate` changed by nurse (wrong date selected from document)
- Patient match rejected and manually selected (matching error)

**Quality thresholds that trigger investigation:**
- Override rate for any field exceeds 20% over a rolling 7-day window: investigate prompt effectiveness for that field.
- Override rate for `documentType` exceeds 10%: indicates classification regression. Rerun reference dataset eval and compare to baseline.
- Override rate for `eventDate` exceeds 30%: expected to be higher due to date ambiguity (Domain FM #3). Investigate only if it increases from baseline.

**F-02: Alert Text Quality Sampling**

Each week, the nurse navigator reviews 10 randomly sampled AI-generated alert descriptions (from non-standard deviations only -- template text is not sampled). The nurse scores each on the 5-point rubric from E-04.

If the weekly average drops below 3.5, or any alert scores 1 (unacceptable), the finding is escalated to the dev team for prompt investigation.

Sampling is weighted toward alerts that were resolved quickly (< 1 hour after creation) -- this may indicate the alert text was unhelpful and the nurse had to independently figure out the correct action.

**F-03 and F-04** are batch analyses run by the dev team using SQL queries against the audit trail and classification log tables. No additional tooling required.

### 7.2 Key Metrics and Alert Thresholds

| Metric | Type | Alert Threshold | Action |
|--------|------|----------------|--------|
| `ai.classification.latency` (p95) | Timer | > 8 seconds | Investigate Claude API latency. Check Anthropic status page. If sustained, consider reducing document text size or switching to Haiku for simple documents. |
| `ai.classification.result{outcome=failure}` rate | Counter ratio | > 10% of calls in 15-min window | Check Claude API errors. If 4xx, investigate prompt changes. If 5xx, wait for circuit breaker recovery. |
| `ai.classification.result{outcome=fallback}` | Counter | > 0 (circuit breaker active) | Circuit breaker is open. Monitor for recovery. If sustained > 5 minutes, check Anthropic API status and API key validity. |
| `ai.alert.latency` (p95) | Timer | > 5 seconds | Alert generation is simpler than classification. High latency here indicates API-level issues. |
| `ai.vision.latency` (p95) | Timer | > 10 seconds | Vision calls are inherently slower. Investigate only if consistently above 10s. |
| `ai.override.rate{field=documentType}` | Gauge (7-day rolling) | > 10% | Classification accuracy may be degrading. Rerun reference dataset eval. Expand few-shot examples for the most-overridden document types. |
| `ai.override.rate{field=eventDate}` | Gauge (7-day rolling) | > 30% | Date extraction is known-hard. Investigate only if trending upward from baseline. |
| `ai.override.rate{field=mrn}` | Gauge (7-day rolling) | > 5% | MRN extraction errors are patient safety risks. Investigate immediately. |
| `anthropic.api.cost.daily` | Counter | > $1.00/day | At pilot scale, >$1/day indicates unexpected volume or a retry loop. |

### 7.4 Sampling Strategy for Human Review

At pilot scale (<500 patients, ~25 docs/day), human review is feasible on a meaningful fraction of AI outputs.

**Document Classification Sampling:**
- **First 2 weeks (launch):** 100% human review. Every AI classification is verified by the nurse before saving. This is already the design (human-in-the-loop), but during launch, actively track and discuss every override in a daily 10-minute standup with the care coordinator.
- **Weeks 3-4:** Review all LOW and MEDIUM confidence classifications. HIGH confidence classifications reviewed only when the nurse overrides.
- **Steady state (month 2+):** Review overrides only. The override itself is the sampling signal -- if the nurse changes a field, that classification enters the review queue.

**Alert Text Sampling:**
- **Weekly:** 10 random AI-generated alert descriptions reviewed by nurse navigator (F-02 flywheel).
- **Weighted sampling:** Over-sample alerts resolved in < 1 hour (potential quality signal) and alerts for cancer types with fewer training examples.

**Escalation to Physician:**
- Any alert description that a nurse flags as "clinically inaccurate" is escalated to the practice physician for review.
- Any classification that results in a wrong-patient match (caught before saving, due to human-in-the-loop) is escalated for root cause analysis.

### 7.5 Dashboard Monitoring View (Pilot Scale)

Rather than building a dedicated AI monitoring dashboard, expose key metrics through the existing admin capabilities:

1. **Spring Boot Actuator `/actuator/metrics`** -- Exposes all Micrometer metrics as JSON. Accessible to the ROLE_ADMIN role.
2. **Weekly summary query** -- SQL query against the classification audit log and alert table, run manually by the dev team during the pilot.

```sql
-- Weekly AI performance summary (run by dev team)
SELECT
    date_trunc('week', created_at) AS week,
    COUNT(*) AS total_classifications,
    COUNT(*) FILTER (WHERE nurse_overrode_type = true) AS type_overrides,
    COUNT(*) FILTER (WHERE nurse_overrode_mrn = true) AS mrn_overrides,
    COUNT(*) FILTER (WHERE nurse_overrode_date = true) AS date_overrides,
    ROUND(AVG(latency_ms)::numeric, 0) AS avg_latency_ms,
    COUNT(*) FILTER (WHERE confidence = 'LOW') AS low_confidence_count,
    COUNT(*) FILTER (WHERE circuit_breaker_fallback = true) AS fallback_count
FROM document_classification_log
WHERE created_at >= NOW() - INTERVAL '4 weeks'
GROUP BY 1
ORDER BY 1 DESC;
```

3. **If dedicated monitoring is needed in V2**, add Grafana (already Docker-friendly) with a PostgreSQL data source pointed at the classification log table. This avoids introducing a new observability platform while providing visual dashboards.
