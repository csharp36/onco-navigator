```java
@ExtendWith(MockitoExtension.class)
class DocumentClassificationServiceTest {

    @Mock private ChatClient classificationClient;
    @Mock private ChatClient.PromptRequest promptRequest;
    @Mock private ChatClient.CallResponse callResponse;

    private DocumentClassificationService service;

    @BeforeEach
    void setUp() {
        // classificationEnabled = true for most tests
        service = new DocumentClassificationService(classificationClient, true);
    }
```

Test cases:
1. `classify_returnsClassification_whenClientSucceeds` -- Mock the ChatClient fluent chain to return a `DocumentClassification` record. Verify the returned object has the expected `documentType`.
2. `classify_returnsNull_whenClientThrowsException` -- Mock ChatClient to throw RuntimeException. Verify `classify()` returns null (no exception propagates).
3. `classify_returnsNull_whenClassificationDisabled` -- Construct service with `classificationEnabled = false`. Verify `classify()` returns null immediately without calling ChatClient.
4. `classifyFallback_returnsNull` -- Call `classifyFallback("text", new RuntimeException("test"))` directly. Verify it returns null.
5. `classify_truncatesLongText` -- Pass text longer than 600,000 chars. Verify the text reaching the mock ChatClient is truncated (use ArgumentCaptor on the user message).

NOTE: The `@CircuitBreaker` annotation is a proxy-based AOP concern -- it only works when Spring creates the proxy. Direct unit tests test the fallback method directly and the core logic. Integration tests (out of scope for this plan) would test the actual circuit breaker behavior with WireMock.

For mocking the ChatClient fluent API chain, you'll need to mock each step:
```java

when(promptRequest.call()).thenReturn(callResponse);

```
