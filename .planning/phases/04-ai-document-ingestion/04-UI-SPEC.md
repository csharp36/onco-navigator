## Interaction Contracts

### Drop Zone

**Dashboard drop zone:**
- Placement: Full-width card below the stat cards section, above Urgent Alerts
- Height: 80px (collapsed), expands to 120px on `dragover`
- Idle state: Dashed border (`border-dashed border-2 border-border`), muted text "Drop a clinical document here, or click to browse", `Upload` lucide icon centered
- Drag-over state: Border changes to `border-ring`, background `bg-muted/50`, text changes to "Release to upload"
- Accepted types: `application/pdf`, `image/jpeg`, `image/png`
- Single file only — if multiple files dropped, reject with inline error: "Please upload one document at a time."
- Click-to-browse: opens `<input type="file" accept=".pdf,.jpg,.jpeg,.png">` — same file types

**Patient detail drop zone:**
- Placement: Inside the Care Events card header row (right side, next to "Record Event" button)
- Appearance: `Button variant="outline" size="sm"` with `Upload` icon and label "Upload Document"
- On click or drop: bypasses patient matching step (patient is already known)
- Drag-over on the Care Events card: entire card highlights with `border-ring`

### Processing Stepper Modal (`DocumentProcessingModal`)

Five steps shown in sequence. Each step shows spinner (pending), check (done), or X (failed).

| Step | Label | Backend operation |
|------|-------|-------------------|
| 1 | Uploading | multipart POST to `/api/documents/upload` |
| 2 | Extracting text | PDFBox / Tesseract / Claude vision pipeline |
| 3 | Classifying document | Claude classification API call |
| 4 | Matching patient | HMAC MRN lookup + name+DOB in-memory search |
| 5 | Ready | Transition to patient match view or pre-filled form |

- Each step transitions after its backend signal (polling or SSE — implementation choice at executor discretion)
- Step in progress: animated `Loader2` lucide icon (spin), label at weight 500
- Step complete: `CheckCircle2` lucide icon in `text-green-600`
- Step failed: `XCircle` lucide icon in `text-destructive`, followed by fallback path (see below)
- Modal is not dismissible while steps 1–4 are in progress (remove close button)
- Modal becomes dismissible (X button re-appears) once step 5 is reached or a failure fallback is shown

**Circuit breaker fallback within stepper:**
- If Step 3 (Classifying) fails (circuit open): show amber banner inside modal — "AI classification is temporarily unavailable. Select the document type manually." + `Select` dropdown of document types (PATHOLOGY_REPORT, RADIOLOGY_REPORT, REFERRAL_LETTER, OPERATIVE_NOTE, LAB_RESULT). Stepper continues to step 4 after selection.
- If Step 4 (Matching) returns no confident match: do not show as failure — show as step complete and transition to `PatientMatchSelector` showing "No automatic match found. Search for a patient or create a new one."

### Patient Match Selector

- Shown inside the same modal after step 5, or as the next modal screen
- High confidence match (>= 85%): show single patient card with `Badge variant="secondary"` labeled "High confidence", "Confirm Patient" primary button, small "Not this patient" link below
- Medium confidence (50–84%): show up to 3 ranked candidates, each a selectable `Card` with patient name, MRN, DOB (decrypted display), confidence `Badge variant="outline"`. "Select" button on each. "Search manually" link.
- Low confidence or no match: show search `Input` (patient name search) and "Create New Patient" `Button variant="outline"` — pre-fills `PatientWizard` with extracted demographics

### Pre-filled Care Event Form

- `QuickAddCareEventDialog` with `prefillData` prop sets `defaultValues` in `react-hook-form`
- Read-only "Source Document" section (above form fields, inside `DialogContent`): document type badge, classification confidence label, thumbnail (128×96px `<img>` of first page if PDF, or original if image), "Preview full document" link that opens `DocumentPreviewPanel`
- Pre-filled fields: `eventType` (mapped from document classification), `eventDate` (extracted from document or today's date), `status` defaults to "COMPLETED"
- Pre-filled fields are editable — user confirms or corrects before saving
- Field values extracted from document are visually distinguished: light `bg-muted/30` input background while pre-filled. Reverts to default input style on user interaction.

### Document Preview Panel (`DocumentPreviewPanel`)

- Trigger: "View Document" button on care event list item (after document is attached)
- Rendered as a `Sheet` (side panel, `side="right"`, `className="w-[600px] sm:max-w-[600px]"`) containing native browser PDF renderer: `<iframe src="{blob-url}" className="w-full h-full" title="Document preview" />`
- Falls back to download link if browser cannot render: "Your browser cannot display this PDF inline. [Download document]"
- "Download" `Button variant="outline"` with `Download` lucide icon always present in sheet header

### Alert Enhancement (Claude-generated descriptions)

No new UI components. Claude-generated text populates `deviationDescription` and `suggestedAction` fields already rendered in `AlertCard`. No visual distinction needed between template-generated and Claude-generated text — both are plain-language descriptions.

**Circuit breaker fallback indicator on alerts:**
- When fallback template text was used (Claude unavailable), add a muted tooltip on the description text: `Tooltip` wrapping the `<p>` with content "AI description unavailable — showing template text"
- Tooltip trigger: `Info` lucide icon (12px, `text-muted-foreground`) inline after the description text
- This is a developer/operational indicator, not a clinical concern

## Accessibility Notes

- Drop zone must be keyboard accessible: `role="button"` + `tabIndex={0}` + `onKeyDown` handler (Enter/Space triggers file picker)
- Drop zone `aria-label`: "Upload clinical document. Drop a file here or press Enter to browse."
- Processing modal while steps 1–4 are in progress: `aria-busy="true"` on modal container
- Patient match candidate cards: `role="group"` with `aria-label="Patient match candidate: {name}"`
- Document preview `<iframe>`: `title="Clinical document preview"` (required for screen reader announcement)
- All `Tooltip` triggers must have `aria-describedby` pointing to tooltip content
