Create the Temporal workflow layer: workflow and activity interfaces, workflow implementations with the signal+timer loop pattern, DTOs for evaluation results and signals, TemporalConfig constants, and PathwayService for starting/signaling workflows.

Purpose: This is the core durable execution engine (INFR-03, INFR-04). The workflow per-patient architecture (D-07) combined with the dual approach of event-driven signals plus daily timer sweeps (D-05) ensures no patient falls through the cracks. Activity interfaces define the contracts that Plan 03 implements.

Output: Two workflow interfaces + implementations, three activity interfaces, two DTOs, one config class, one service class.

**PatientPathwayWorkflowImpl.java** -- Core workflow implementation (per D-05, D-07, D-08, D-09):
- Package: `com.onconavigator.workflow`
- No Spring annotations (workflow classes run in Temporal's context, not Spring's)
- Fields:
  - `private boolean signalReceived = false;`
  - `private boolean deactivated = false;`
  - `private boolean pathwayComplete = false;`
  - `private UUID patientId;`
- Activity stubs created via `Workflow.newActivityStub()`:
  - `PathwayEvaluationActivity evaluationActivity` with ActivityOptions: startToCloseTimeout = 2 minutes, retryOptions = 3 attempts, 5s initial interval, 2.0 backoff coefficient, maxInterval 1 minute. Do NOT retry `IllegalArgumentException` (add to `setDoNotRetry`).
  - `AlertGenerationActivity alertActivity` with same retry policy.
- `monitorPathway(UUID patientId, String cancerType)` implementation:
  ```
  this.patientId = patientId;
  while (!deactivated && !pathwayComplete) {
      signalReceived = false;
      // D-05: dual approach -- wait 24h, wake early on signal
      Workflow.await(Duration.ofHours(24), () -> signalReceived || deactivated);
      if (deactivated) {
          evaluationActivity.closeOpenAlerts(patientId);
          break;
      }
      PathwayEvaluationResult result = evaluationActivity.evaluate(patientId);
      pathwayComplete = result.allStepsComplete();
  }
  ```
- `careEventChanged(UUID careEventId)`: sets `signalReceived = true;`
- `deactivatePatient(String reason)`: sets `deactivated = true;`
- `getPathwayStatus()`: returns `pathwayComplete ? "COMPLETE" : (deactivated ? "DEACTIVATED" : "MONITORING")`
- CRITICAL: NO database access, NO `System.currentTimeMillis()`, NO `new Random()`, NO `Thread.sleep()` -- workflow code must be deterministic. Use ONLY `Workflow.*` APIs for time and randomness.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Information Disclosure | PatientPathwayWorkflow.monitorPathway | mitigate | Method signature accepts only `UUID patientId` and `String cancerType`. No PHI fields. Enforced by interface contract. Code review: grep workflow method signatures for String params that could carry PHI. |
| T-02-06 | Information Disclosure | PatientPathwayWorkflow.careEventChanged | mitigate | Signal carries only `UUID careEventId`. Activity fetches event details from encrypted DB. |
| T-02-07 | Elevation of Privilege | PathwayService.startPathwayMonitoring | mitigate | PathwayService methods are internal Spring beans. Phase 3 REST controllers will enforce RBAC via @PreAuthorize before calling PathwayService. This phase builds the service; access control is at the controller layer. |
| T-02-08 | Tampering | Workflow signal injection | mitigate | Signals carry only UUID values. Alert text comes from pathway templates (admin-controlled JSONB in DB), never from signal payloads. An attacker who sends a rogue signal can only trigger a re-evaluation, which is idempotent. |
| T-02-09 | Denial of Service | Workflow creation per patient | accept | One workflow per patient at pilot scale (dozens of patients). WorkflowIdReusePolicy.ALLOW_DUPLICATE prevents ID collisions. Not a concern at V1 scale. |
