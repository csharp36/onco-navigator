**PathwayEvaluationActivityImpl.java** -- The core deviation detection engine:
- Package: `com.onconavigator.activity`
- Annotations: `@Component` (Spring bean discovery for Temporal auto-config)
- Javadoc: "Evaluates a patient's care pathway against the template, detecting missing, delayed, and out-of-order events. All database access happens here -- the workflow passes only the patient UUID. PHI note: This activity logs only patient UUIDs and step names. Never log patient names, DOBs, or MRNs."
- Constructor injection (6 dependencies):
  ```java
  private final PatientRepository patientRepository;
  private final CareEventRepository careEventRepository;
  private final PathwayTemplateRepository templateRepository;
  private final PhysicianOverrideRepository overrideRepository;
  private final AlertGenerationActivity alertGenerationActivity; // Call alert generation as separate activity for clean separation
  private final ObjectMapper objectMapper;
  private static final Logger log = LoggerFactory.getLogger(PathwayEvaluationActivityImpl.class);
  ```

  IMPORTANT: Re-evaluate whether calling AlertGenerationActivity from within PathwayEvaluationActivity is appropriate. The cleaner approach is to have PathwayEvaluationActivityImpl directly use `AlertRepository` and `AlertGenerationActivityImpl` logic inline, OR have the workflow call AlertGenerationActivity separately after evaluate() returns detected deviations. The recommended approach: PathwayEvaluationActivityImpl detects deviations AND creates alerts directly (using AlertRepository), returning PathwayEvaluationResult. This avoids activity-calling-activity complexity.

  Revised constructor injection (7 dependencies):
  ```java
  private final PatientRepository patientRepository;
  private final CareEventRepository careEventRepository;
  private final AlertRepository alertRepository;
  private final PathwayTemplateRepository templateRepository;
  private final PhysicianOverrideRepository overrideRepository;
  private final ObjectMapper objectMapper;
  private static final Logger log = LoggerFactory.getLogger(PathwayEvaluationActivityImpl.class);
  ```

  `evaluate(UUID patientId)` implementation:

  1. Fetch patient: `patientRepository.findById(patientId).orElseThrow(() -> new IllegalArgumentException("Patient not found: " + patientId));`
  2. Fetch care events: `careEventRepository.findByPatient_IdOrderByEventDateDesc(patientId);`
  3. Fetch pathway template: `templateRepository.findByCancerType(patient.getCancerType()).orElseThrow(() -> new IllegalArgumentException("No pathway template for: " + patient.getCancerType()));`
  4. Deserialize JSONB: `List<PathwayStep> steps = objectMapper.readValue(template.getTemplateData(), new TypeReference<List<PathwayStep>>() {});`
  5. Build helper maps:
     - `Map<CareEventType, List<CareEvent>> eventsByType` -- group care events by eventType
     - `Map<String, CareEvent> completedByStepId` -- for each step, find the COMPLETED care event (match by eventType, any COMPLETED event for that type counts)
     - `Set<String> completedStepIds` -- step IDs with a COMPLETED care event

  6. For each PathwayStep in `steps`, evaluate in order:

     **a. Check physician override (PATH-08):**
     - `if (overrideRepository.existsByPatientIdAndPathwayStepId(patientId, step.stepId()))` -- skip this step, no alert
     - Log: `log.debug("Override exists for patient {} step {}, skipping evaluation", patientId, step.stepId());`

     **b. Check if step is completed:**
     - Match care events where `eventType == step.eventType()` AND `status == CareEventStatus.COMPLETED`
     - If completed: mark in `completedStepIds`, continue to next step

     **c. Detect OUT-OF-ORDER (PATH-05):**
     - Check if ANY care event exists for this step (status = SCHEDULED, COMPLETED, or PENDING) but one or more of its `prerequisites` are NOT in `completedStepIds`
     - Detection rule: Find care events matching `step.eventType()` with status in [SCHEDULED, COMPLETED, PENDING]. If such an event exists AND any prerequisite step ID is NOT in `completedStepIds`, this is out-of-order.
     - Alert text: use `step.alertText()` from template
     - Suggested action: use `step.suggestedAction()` from template
     - Create alert if no duplicate (dedup check): `!alertRepository.existsByPatientIdAndPathwayStepNameAndStatus(patientId, step.name(), AlertStatus.OPEN)`
     - If creating alert: build Alert entity, set fields, save via `alertRepository.save()`

     **d. Detect MISSING or DELAYED (PATH-03, PATH-04):**
     - Only check for required steps (`step.required() == true`)
     - Calculate anchor date based on `step.anchorType()`:
       - `DIAGNOSIS_DATE`: `patient.getDiagnosisDate()`
       - `PREVIOUS_STEP`: the `eventDate` of the COMPLETED care event for the previous step (by stepNumber - 1). If previous step is not completed, skip delay check (can't be delayed if prerequisite isn't done).
       - `SPECIFIC_STEP`: the `eventDate` of the COMPLETED care event for the step identified by `step.anchorStepId()`. If that step is not completed, skip delay check.
     - Calculate elapsed days: `ChronoUnit.DAYS.between(anchorDate, LocalDate.now())`
     - If no care event exists for this step's eventType (no COMPLETED, no SCHEDULED, no PENDING):
       - If `elapsed > step.windowDays()`: generate MISSING_EVENT alert (PATH-03 -- also covers delayed, but MISSING is more accurate when no event exists at all)
     - If a care event exists but not COMPLETED and `elapsed > step.windowDays()`:
       - Generate DELAYED_EVENT alert (PATH-04)
     - Dedup check before each alert creation (PATH-06)

  7. Determine `allStepsComplete`: all step IDs are in `completedStepIds`
  8. Build `List<String> alertsGenerated` -- summary strings like "MISSING_EVENT: step Surgeon Consultation for patient {patientId}"
  9. Log evaluation result (PATH-07): `log.info("PATHWAY_EVALUATION: patient={} stepsEvaluated={} alertsGenerated={} allComplete={}", patientId, steps.size(), alertsGenerated.size(), allStepsComplete);`
  10. Return `new PathwayEvaluationResult(allStepsComplete, alertsGenerated)`

  `closeOpenAlerts(UUID patientId)` implementation:
  - `List<Alert> openAlerts = alertRepository.findByPatientIdAndStatus(patientId, AlertStatus.OPEN);`
  - For each alert: set status to RESOLVED, set resolvedAt to OffsetDateTime.now(), set resolutionNotes to "Patient deactivated -- workflow cancelled"
  - Save all: `alertRepository.saveAll(openAlerts);`
  - Log: `log.info("Closed {} open alerts for deactivated patient {}", openAlerts.size(), patientId);`

- Package: `com.onconavigator.activity`
- Annotations: `@Component`

- `private static final Logger log = LoggerFactory.getLogger(SweepActivityImpl.class);`
- Javadoc: "Daily sweep activity that ensures all active patients have a running pathway workflow. Queries active patients, checks which ones have a running workflow, and starts workflows for any that are missing."

  `findAndStartMissingWorkflows()` implementation:
  1. `List<Patient> activePatients = patientRepository.findByStatus(PatientStatus.ACTIVE);`
  2. Counters: `int started = 0; int skipped = 0;`
  3. For each active patient:
     a. Build expected workflow ID: `TemporalConfig.PATHWAY_WORKFLOW_ID_PREFIX + patient.getId()`
     b. Try to describe the workflow via WorkflowClient to check if it's running:
        ```java
        try {
            WorkflowStub existingWorkflow = workflowClient.newUntypedWorkflowStub(workflowId);
            // If we can describe it without exception, it exists
            // Use WorkflowClient to check -- attempt to describe the execution
            workflowClient.newUntypedWorkflowStub(workflowId).getResult(0, java.util.concurrent.TimeUnit.MILLISECONDS, Void.class);
            skipped++;
        } catch (Exception e) {
            // Workflow not found or completed -- start a new one
        }
        ```

        ACTUALLY, the simpler and correct approach: Try to start the workflow with `WORKFLOW_ID_REUSE_POLICY_REJECT_DUPLICATE`. If it throws `WorkflowExecutionAlreadyStartedError`, the workflow already exists. If it succeeds, we just started a missing workflow.

        Revised approach -- use the existing `PathwayService` but handle the already-started case:
        ```java
        try {
            WorkflowOptions options = WorkflowOptions.newBuilder()
                .setWorkflowId(TemporalConfig.PATHWAY_WORKFLOW_ID_PREFIX + patient.getId())
                .setTaskQueue(TemporalConfig.TASK_QUEUE)
                .setWorkflowIdReusePolicy(
                    WorkflowIdReusePolicy.WORKFLOW_ID_REUSE_POLICY_REJECT_DUPLICATE)
                .build();
            PatientPathwayWorkflow workflow = workflowClient.newWorkflowStub(
                PatientPathwayWorkflow.class, options);
            WorkflowClient.start(workflow::monitorPathway, patient.getId(), patient.getCancerType().name());
            started++;
            log.info("SWEEP_STARTED: Started missing workflow for patient {}", patient.getId());
        } catch (WorkflowExecutionAlreadyStarted e) {
            skipped++;
        }
        ```
        Import: `io.temporal.client.WorkflowExecutionAlreadyStarted`

     c. This is the correct pattern -- try-to-start-with-reject is idempotent and avoids race conditions.
  4. Log: `log.info("DAILY_SWEEP: activePatients={} workflowsStarted={} alreadyRunning={}", activePatients.size(), started, skipped);`

  NOTE: SweepActivityImpl injects `WorkflowClient` directly (not `PathwayService`) because:
  - It needs `REJECT_DUPLICATE` policy (not `ALLOW_DUPLICATE` which PathwayService uses)
  - It needs to catch `WorkflowExecutionAlreadyStarted` specifically
  - It avoids circular dependency (PathwayService -> SweepActivity -> PathwayService)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Activity -> PostgreSQL | Activities read patient data from encrypted DB; only UUIDs cross back to Temporal |
| Activity -> AlertRepository | Alert creation writes non-PHI clinical process text to alerts table |
| SweepActivity -> WorkflowClient | Sweep starts workflows for active patients; only UUIDs passed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Information Disclosure | PathwayEvaluationActivityImpl | mitigate | Activity logs only patient UUIDs and step names. PHI fields (firstName, lastName, etc.) are never referenced in log statements. Alert deviationDescription uses clinical process text from pathway template, not patient data. |
| T-02-11 | Tampering | Alert dedup bypass | mitigate | `existsByPatientIdAndPathwayStepNameAndStatus` is a DB query that cannot be bypassed from client side. Dedup is enforced at the repository layer, not in application logic alone. |
| T-02-12 | Denial of Service | SweepActivity starting all patient workflows | accept | At V1 pilot scale (dozens of patients), starting all missing workflows is not a resource concern. The try-to-start-with-reject pattern is idempotent -- duplicate starts are caught and skipped. |
| T-02-13 | Repudiation | PATH-07 evaluation logging | mitigate | Every evaluation produces a structured log message (PATHWAY_EVALUATION) with patient UUID, step count, alert count, and completion status. Temporal event history also records every activity execution with timestamps. |
| T-02-14 | Elevation of Privilege | Physician override suppression | mitigate | Override records are checked via `existsByPatientIdAndPathwayStepId` before alert generation. Only records in the physician_overrides table (created by authorized users with @Audited trail) can suppress alerts. |
</threat_model>
