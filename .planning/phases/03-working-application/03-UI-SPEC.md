## Typography

All sizes in pixels (rem conversions for Tailwind classes shown):

| Role | Size | Tailwind | Weight | Weight Name | Line Height | Usage |
|------|------|----------|--------|-------------|-------------|-------|
| Body | 14px | `text-sm` | 400 | Regular | 1.5 | Form labels, table cell text, nav items, card body copy |
| UI Label | 16px | `text-base` | 400 | Regular | 1.5 | Input field text, dialog body, resolution notes |
| Subheading | 20px | `text-xl` | 600 | Semibold | 1.3 | Card titles, section headings, modal titles |
| Heading | 28px | `text-3xl` | 600 | Semibold | 1.2 | Page titles (Dashboard, Patients, Alerts) |

**Constraints:**
- Only 4 sizes and exactly 2 weights are permitted: Regular (400) and Semibold (600)
- 600 Semibold serves both Subheading (20px) and Heading (28px) — size differential provides sufficient visual hierarchy without a third weight
- `text-muted-foreground` class applied to secondary labels, timestamps, help text
- `tracking-tight` applied to all headings (matches existing Dashboard h1 pattern from routes/index.tsx)

## Interaction Patterns

### Two-Step Patient Wizard
- Step indicator at top: two circles connected by a line. Active step = `bg-primary text-primary-foreground`. Inactive = `bg-muted text-muted-foreground`.
- Validation occurs on Next click (step 1) and Save click (step 2). Invalid fields show red border (`aria-invalid`) and inline error text in `text-destructive text-xs` below field.
- Back button on step 2 restores step 1 form data from react-hook-form state.
- On successful save: redirect to `/patients/{id}` (D-02 from CONTEXT.md).

### Alert Card Layout
- Card with `border-l-4` left stripe in severity color (4px, full card height).
- Card header: severity `Badge` + patient name as `Link` (underline on hover) + MRN in `text-muted-foreground`.
- Card body: pathway step name (semibold), deviation description, suggested action.
- Card footer: time elapsed (`text-muted-foreground text-sm`) + [View] `Button` (outline sm) + [Resolve] `Button` (default sm).
- Cards are grouped by severity with a section label: "Overdue", "Missing", "Out of Order". Labels use `text-sm font-semibold text-muted-foreground uppercase tracking-wide`.

### Alert Resolution Modal
- Opened by [Resolve] button on alert card.
- Modal header: "Resolve Alert" heading + alert details summary (patient name, step, severity badge) — read-only.
- Resolution notes `Textarea` is required. Resolve button disabled until at least 10 characters entered.
- On successful resolve: modal closes, TanStack Query invalidates `['alerts']` and `['alert-count']` keys. Alert disappears from queue (optimistic update pattern).

### Pathway Visualization (Vertical Stepped List)
- Each step is a row: `[icon] [step number] [step name] — [status line]`
- Icons from lucide-react: `CheckCircle2` (green, completed), `Clock` (amber, overdue/missing), `AlertTriangle` (red, alert active), `Circle` (muted, upcoming).
- Status line: "Completed 2026-03-15 (Day 8 of 14-day window)" or "Overdue — 6 days past window" or "Waiting for Step 2".
- Steps with active alerts show a subtle `bg-amber-50` row background (light amber tint — defined inline, not a token).

### Navigation Alert Badge
- `<Bell>` icon in nav sidebar gains a red pill badge showing open alert count.
- Badge positioned absolute, top-right of the icon. Renders only when count > 0.
- Count fetched via `/api/alerts/count` endpoint with TanStack Query `refetchInterval: 30_000` (D-12 from CONTEXT.md).
- Badge max display: "99+" for counts over 99.

### Data Loading States
- All list and detail pages show `Skeleton` during initial load.
- Patient list: 5 skeleton rows at 48px height.
- Alert queue: 3 skeleton cards at 100px height.
- Patient detail: skeleton header + 2-column skeleton content.
- Error states (API failures): replace content with error message card (see Copywriting Contract).

### Patient Deactivation (DATA-04)
- "Deactivate Patient" is a destructive action. Uses a `Dialog` confirmation (not inline confirmation).
- Button: `variant="destructive"` on patient detail page.
- Confirmation dialog has a required reason select: "Deceased", "Transferred to Another Practice", "Withdrawn from Care".

## Visual Focal Points

**Dashboard primary screen:** The primary visual anchor on the Dashboard is the Urgent Alerts card list. The stat cards (Open Alerts, Active Patients, On-Track Patients) at the top orient the nurse to overall state; the eye then moves immediately to the alert card list below, which is where action is required. The "Open Alerts" stat card number is the secondary focal point — it is rendered at 28px Semibold to draw attention before the user scans the card list.

**Alert Queue:** The severity badge on each alert card (`border-l-4` left stripe in severity color) is the focal point — color immediately communicates urgency before the user reads any text.

**Patient Detail:** The patient demographics header bar and active alert indicators on pathway steps are the focal points — the nurse needs to confirm patient identity before acting.
