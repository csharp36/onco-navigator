**PatientWizard.tsx** — two-step wizard per D-01:

Step indicator at top: two numbered circles (32px) connected by a line. Active step: `bg-primary text-primary-foreground`. Inactive: `bg-muted text-muted-foreground`.

Step 1 — Demographics (per D-01):
- Fields: firstName (Input, required), lastName (Input, required), dateOfBirth (Input type="date", required), mrn (Input, required)
- Zod v4 schema with `{ error: '...' }` syntax (NOT `{ message: '...' }`):
```typescript
const step1Schema = z.object({
  firstName: z.string().min(1, { error: 'First name is required.' }),
  lastName: z.string().min(1, { error: 'Last name is required.' }),
  dateOfBirth: z.string().min(1, { error: 'Date of birth is required.' }),
  mrn: z.string().min(1, { error: 'MRN is required.' }),
});
```
- Uses its own `useForm` instance with `zodResolver(step1Schema)`
- "Next: Clinical Details" button validates step 1, advances to step 2
- Invalid fields show red border (`aria-invalid`) and error text in `text-destructive text-xs`

Step 2 — Clinical Info (per D-01, DATA-01):
- Fields: cancerType (Select with BREAST/LUNG/COLORECTAL options), cancerStage (Input, required), diagnosisDate (Input type="date", required), assignedNavigator (Input, optional — freetext name per Open Question 3 resolution), treatingPhysician (Input, optional)
- "Assigned Navigator" is a freetext Input (not a dropdown) because V1 pilot has no user directory. The care coordinator types the navigator's name. The field is optional (nullable on backend).
- Zod v4 schema:
```typescript
const step2Schema = z.object({
  cancerType: z.string().min(1, { error: 'Cancer type is required.' }),
  cancerStage: z.string().min(1, { error: 'Cancer stage is required.' }),
  diagnosisDate: z.string().min(1, { error: 'Diagnosis date is required.' }),
  assignedNavigator: z.string().optional(),
  treatingPhysician: z.string().optional(),

```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser rendering | Decrypted PHI (patient name, MRN, DOB) rendered in DOM — acceptable for authenticated users |
| Client-side role checks | hasRole() used for conditional UI rendering — authorization enforced server-side via @PreAuthorize |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-13 | Elevation of Privilege | Client-side hasRole for button visibility | mitigate | hasRole() hides UI elements but all write operations are enforced server-side by @PreAuthorize on controllers. Frontend role checks are UX convenience, not security boundaries. |
| T-03-14 | Tampering | Zod v4 form validation | mitigate | Client-side validation with Zod schemas provides immediate user feedback; server-side @Valid + Bean Validation is the authoritative check. Both must pass for data to be accepted. |
| T-03-15 | Information Disclosure | PHI in browser DOM | accept | Patient name, MRN, DOB displayed to authenticated, authorized users on the patient detail page. This is the intended use case — no mitigation needed beyond existing JWT auth. |
</threat_model>
