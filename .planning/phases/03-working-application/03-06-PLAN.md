<task type="checkpoint:human-verify" gate="blocking">
  <what-built>
Complete Phase 3 working application: REST API (patient CRUD, care events, alert management, dashboard stats), React frontend (patient wizard, patient list, patient detail with pathway visualization, alert queue with resolve modal, dashboard with stat cards, sidebar alert badge with 30-second polling).
  </what-built>
  <how-to-verify>
**Prerequisites:**
1. Start the full stack: `docker compose up -d` (PostgreSQL, Keycloak, Temporal)
2. Run the backend: `./mvnw spring-boot:run` from project root
3. Run the frontend: `cd frontend && npm run dev`
4. Log in at http://localhost:5173 with a Keycloak user

**Flow 1 — Patient Creation (DATA-01, D-01, D-02):**
1. Click "Patients" in sidebar
2. Click "Add Patient" button
3. Fill wizard Step 1: first name, last name, DOB, MRN
4. Click "Next: Clinical Details" — verify step 1 validates (try empty fields)
5. Fill wizard Step 2: cancer type (select), stage, diagnosis date
6. Click "Enroll Patient"
7. Verify: redirected to patient detail page showing the new patient's demographics, pathway steps, and empty care events list

**Flow 2 — Care Event Recording (DATA-02, D-03):**
1. On the patient detail page, click "Record Event"
2. Fill in event type, date, status, optional notes
3. Click "Save Event"
4. Verify: care event appears in the care events list on the right
5. Go to Patients list, find the patient row, click the "Record Event" button
6. Verify: quick-add dialog opens, fill and save
7. Verify: both entry points work

**Flow 3 — Alert Queue (ALRT-01, ALRT-02, ALRT-04):**
1. Click "Alerts" in sidebar
2. Verify: alerts are grouped by severity (OVERDUE section first, then MISSING, then OUT OF ORDER)
3. Verify: each alert card shows patient name (clickable), MRN, severity badge with color, step name, description, suggested action, time elapsed
4. Click a patient name on an alert card — verify: navigates to patient detail page
5. Click [Resolve] on an alert — verify: modal opens with alert details
6. Try submitting with fewer than 10 characters — verify: validation error
7. Enter 10+ character notes, click "Resolve Alert"
8. Verify: modal closes, alert disappears from queue

**Flow 4 — Dashboard (D-10, D-11, D-12, ALRT-05):**
1. Click "Dashboard" in sidebar (or go to /)
2. Verify: three stat cards show Open Alerts count, Active Patients, On-Track Patients
3. Verify: top urgent alerts are displayed below stat cards
4. Verify: sidebar "Alerts" item shows a red count badge (if alerts exist)
5. Navigate to different pages — verify: badge count persists on sidebar
6. Wait 30+ seconds — verify: counts update without manual refresh

**Flow 5 — Patient Deactivation (DATA-04):**
1. Go to a patient detail page
2. Click "Deactivate Patient" (should be red/destructive button)
3. Verify: confirmation dialog with reason selection
4. Select a reason, confirm
5. Verify: patient status changes, pathway monitoring stops

**Flow 6 — Responsive Layout (SEC-07):**
1. Resize browser to tablet width (~768px)
2. Verify: patient detail page stacks vertically (pathway above care events)
3. Verify: no horizontal scrolling on any page
4. Verify: sidebar collapses to hamburger menu
  </how-to-verify>
  <resume-signal>Type "approved" if all flows work correctly, or describe any issues found.</resume-signal>
