> Locks framework selection, implementation guidance, and evaluation strategy before planning begins.

---

## 1. System Classification

**System Type:** Extraction

**Description:**
When a clinical document (MD notes, orders, nurse notes) is uploaded for a patient, Claude AI extracts ordered/planned care events and proposes them as new pathway steps with dependency edges. Extracted steps appear as PROPOSED in the patient's DAG pathway. A nurse navigator must confirm each step before it becomes active in the evaluation engine. The system must never auto-confirm AI-extracted steps — human-in-the-loop is non-negotiable for clinical safety.

**Critical Failure Modes:**
1. **Hallucinated care events** — Claude proposes steps that are not mentioned or implied in the document text (false positives that waste nurse time and erode trust)
2. **Invalid CareEventType enum values** — Claude returns event types not in the application's enum, causing deserialization failures or silent data corruption
3. **Auto-activation without nurse confirmation** — Steps bypass the PROPOSED status and become active in the DAG evaluation, triggering alerts for steps the nurse never approved
4. **Missed critical steps** — Claude fails to extract a clearly stated treatment plan item from the document (false negatives that defeat the purpose of the system)
5. **Invalid edge inference** — Claude proposes dependency edges that create cycles in the DAG or connect to nonexistent steps, corrupting the pathway graph

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Medical Oncology — care coordination and pathway monitoring at an outpatient oncology practice

**User Population:** Nurse navigators who manage patient care coordination across multiple external facilities (surgery, radiology, pathology). Users are trained clinical professionals, not data entry clerks — they will immediately recognize clinically implausible AI output.

**Stakes Level:** High

**Output Consequence:** Each AI-extracted step that a nurse confirms becomes an active node in the patient's care pathway DAG. The pathway engine then monitors for missing, delayed, or out-of-sequence events and fires alerts. A confirmed hallucinated step generates false alerts that erode trust. A missed critical step means a required treatment milestone is not monitored — the patient can fall through the cracks with no alert. Neither failure mode causes direct patient harm in V1 (the nurse confirms before anything activates), but both degrade the core value proposition: systematic, comprehensive pathway monitoring.

### What Domain Experts Evaluate Against

```
Dimension: Document fidelity — steps extracted, not invented
Good: Every proposed step can be traced to a specific sentence or phrase in the uploaded document.
      "Radiation therapy — 6 weeks, 5 days/week" in the note becomes one proposed step; no steps appear
      that are plausible-but-absent, such as a routine bone scan that was not mentioned.
Bad:  Claude proposes "CT follow-up at 3 months" because it is standard of care for this cancer type,
      but the phrase does not appear in this document. The nurse has no way to verify the source.
Stakes: Critical
Source: Clinical NLP practitioner standard — "hallucinations" are the most cited limitation in oncology
        LLM extraction literature (38% of reviewed studies flagged accuracy/hallucination as primary
        concern). Unsourced clinical steps have no evidentiary basis and cannot be audited.
```

```
Dimension: Completeness — no treatment plan items silently dropped
Good: A note listing "1) biopsy, 2) surgical consultation, 3) staging CT, 4) multidisciplinary tumor
      board review" produces four proposed steps. If one is missed, the nurse notices during review
      because the numbered list is visible in the source document.
Bad:  The AI returns only two of the four items because the third was stated parenthetically or used
      an abbreviation ("MDT review") not recognized as a care event.
Stakes: High
Source: Production oncology NLP deployments consistently show performance degradation on abbreviated
        or parenthetical clinical language (JMIR Cancer 2025 scoping review). A missed step is a
        monitoring blind spot — the pathway engine cannot alert on something it does not know to watch.
```

```
Dimension: Sequencing and dependency accuracy
Good: If the note states "biopsy results must be reviewed before treatment planning meeting," the
      proposed edge runs biopsy → treatment planning, not the reverse. If the note implies no
      specific order between two steps, no edge is proposed between them.
Bad:  Claude infers a dependency from general oncology knowledge ("CT scan should precede surgery")
      rather than from what the document states, or proposes a cycle (A→B→A) that makes the DAG
      structurally invalid.
Stakes: High
Source: Oncologist review of this system (docs/Pathway-Template-Review-Worksheet.md, PW-ALL-002):
        "each patient needs unique sequence." Imposing standard-of-care sequencing when the physician
        wrote something different overrides the physician's intent.
```

```
Dimension: Clinical term normalization — event types the system can actually track
Good: "Post-op follow-up appointment" is mapped to FOLLOW_UP_VISIT. "Chemotherapy infusion #1"
      is mapped to CHEMOTHERAPY. The mapping is recognizable to the nurse as correct.
Bad:  Claude invents a new event type label ("CHEMO_INFUSION_CYCLE_1") that does not exist in the
      CareEventType enum, causing a deserialization failure, or maps "tumor board review" to
      PHYSICIAN_CONSULTATION when the practice tracks these as distinct event types.
Stakes: Critical
Source: Structured extraction systems in clinical settings fail silently when output schema is
        unconstrained. The enum constraint is a hard system boundary — invalid values corrupt
        downstream pathway evaluation.
```

```
Dimension: Noise rejection — administrative content excluded
Good: A document with clinical orders embedded in a scheduling confirmation email yields only the
      clinical care steps (biopsy, pathology review). The appointment logistics (fax number,
      insurance pre-authorization instructions) are not extracted as pathway steps.
Bad:  Claude proposes "Insurance pre-authorization" or "Fax results to referring physician" as
      pathway steps because they appear in the document as action items.
Stakes: Medium
Source: Practitioner knowledge — oncology clinical documents (especially referral letters and
        operative notes) routinely contain administrative boilerplate alongside clinical orders.
        Administrative noise inflates the nurse's confirmation workload and trains them to reject
        AI proposals reflexively.
```

### Known Failure Modes in This Domain

1. **Negation misidentification** — Clinical notes routinely state what is NOT planned ("patient is NOT a surgical candidate," "radiation is contraindicated"). LLMs, including general-purpose models, show documented difficulty with clinical negation, extracting negated items as positive proposals. A nurse navigator reviewing "RADIATION THERAPY — CONTRAINDICATED" expects zero radiation steps proposed; receiving one is an immediate trust-breaking error.

2. **Abbreviation and shorthand collapse** — Oncology notes use dense abbreviations: "XRT" (radiation therapy), "MDT" (multidisciplinary team), "NACT" (neoadjuvant chemotherapy), "BSO" (bilateral salpingo-oophorectomy). General-purpose models trained on general text may map uncommon abbreviations to the wrong care event type or fail to extract them entirely.

3. **Multi-step conflation** — A single sentence can describe multiple distinct care events ("surgery followed by adjuvant chemotherapy and radiation"). Models frequently extract this as one step rather than three, producing a single vague "surgery/chemo/radiation" proposal that cannot be mapped to a CareEventType and provides no monitoring granularity.

4. **Long document degradation** — Operative notes and multi-visit physician notes can run 2,000–5,000 tokens. Research across oncology NLP deployments shows consistent performance degradation as document length increases, with relevant treatment plan items buried in narrative context missed at higher rates than items in structured list format.

### Regulatory / Compliance Context

**HIPAA — Directly Applicable**

Clinical document full text (extracted via Apache Tika in Phase 4) is sent to Claude via the Anthropic API. This transmission constitutes disclosure of ePHI to a business associate. Key obligations:

- A signed BAA with Anthropic must be in place before any real patient documents are processed. The feature flag (`onconavigator.ai.step-extraction.enabled`) must remain `false` until BAA is executed.
- The Anthropic API call must occur over TLS (enforced by the SDK; no additional configuration required).
- The 2025 HHS proposed HIPAA Security Rule update removes the "addressable vs. required" distinction for safeguards — encryption, access logging, and risk analysis coverage of AI tools are now effectively mandatory.
- Minimum necessary standard applies: the prompt should send only the document text and the list of existing step event types needed for dedup context — not broader patient record data.

**No additional regulations identified** for this specific deployment context (single-practice V1 pilot, US jurisdiction, no FDA Software as a Medical Device classification applies because the AI output requires nurse confirmation before clinical use — this keeps it in the clinical decision support category, not autonomous clinical action).

## 2. Framework Decision

**Selected Framework:** Spring AI 1.1.0 (spring-ai-starter-model-anthropic)

**Version:** 1.1.0 (GA, via Spring Boot 3.5 BOM)

**Rationale:**
This is a single-model structured extraction call inside a Java Spring Boot application. The project already has `DocumentClassificationService` as a working reference implementation using the identical pattern: `ChatClient` `@Qualifier` bean, `@CircuitBreaker` on "claude-api", feature flag via `@Value`, null-return fallback, and structured JSON output. Phase 6 adds one new `ChatClient` bean (`stepExtractionClient`) in `AiClientConfig` and a `StepExtractionService` following that exact template. Introducing any external Python or TypeScript agent framework would add a polyglot runtime, new infrastructure, and a new failure domain for a problem the existing stack solves with a single API call.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Spring AI BeanOutputConverter | Not ruled out — viable upgrade path if extraction JSON schema grows complex. Spring AI's BeanOutputConverter binds Claude's structured output directly to a Java record, eliminating manual Jackson parsing while remaining entirely within the existing Spring AI dependency. |
| LangChain / LangGraph / CrewAI / LlamaIndex | All are Python/TypeScript frameworks. Introducing a polyglot runtime and new infrastructure into a Java Spring Boot application for a single extraction call is an anti-pattern. |
| Direct Anthropic Java SDK | Spring AI wraps this already. Using the raw SDK would lose the ChatClient abstraction, making future model changes harder. |

**Vendor Lock-In Accepted:** Partial — committed to Anthropic/Claude for AI model, but Spring AI's ChatClient abstraction provides a model-swap escape hatch if needed.

### 3.3 Minimal Entry Point for Extraction

`ChatClient.entity(Class)` is the extraction primitive. Spring AI generates a JSON schema from the target Java record, appends format instructions to the prompt, calls Claude, and deserializes the response — all in one call:

```java
// Returns a strongly-typed Java record from Claude's structured JSON output.
// Spring AI handles schema generation, format instruction injection, and Jackson deserialization.
ExtractionResult result = stepExtractionClient.prompt()
        .user(u -> u.text(ExtractionPrompts.USER_TEMPLATE)
                     .param("documentText", truncated)
                     .param("existingSteps", existingStepsJson))
        .call()
        .entity(ExtractionResult.class);
```

For a response containing an array at the root, use `ParameterizedTypeReference`:

```java
List<ProposedStep> steps = stepExtractionClient.prompt()
        .user(u -> u.text(ExtractionPrompts.USER_TEMPLATE)
                     .param("documentText", truncated)
                     .param("existingSteps", existingStepsJson))
        .call()
        .entity(new ParameterizedTypeReference<List<ProposedStep>>() {});
```

The top-level wrapper record (`ExtractionResult` containing `List<ProposedStep>` plus `alreadyCoveredEventTypes`) is preferred over the bare list form. A named root object produces more specific schema validation errors on retry, and the wrapper field `alreadyCoveredEventTypes` is needed for the "already covered" transparency display (D-10).

### 3.4 Core Abstractions

| Abstraction | Role in Phase 6 | Notes |
|---|---|---|
| `ChatClient` | Stateless fluent API for a single-turn Claude call. One bean per use case, created via `ChatClient.builder(chatModel)`. | Never call `.defaultSystem()` on the auto-configured `ChatClient.Builder` singleton — it mutates shared state across all beans that use the same builder. Always call `ChatClient.builder(chatModel)` (static factory, fresh builder) per `@Bean` method. Enforced in `AiClientConfig` CR-05. |
| `AnthropicChatOptions` | Per-bean overrides for `temperature`, `maxTokens`, and prompt caching strategy. | `maxTokens` has no default when omitted from the builder — Claude will generate until it decides to stop, causing unbounded cost and latency. Always set it explicitly. |
| `.entity(Class)` / `.entity(ParameterizedTypeReference)` | Structured output — generates JSON schema from a Java record, appends format instructions to the prompt, deserializes Claude's response via Jackson. | Requires `@JsonProperty(required = true)` on mandatory fields. Without it, the generated schema marks fields as optional and Claude may omit them silently. The absence is not an error at the Spring AI layer — it surfaces as a `NullPointerException` in downstream code. |
| `StructuredOutputValidationAdvisor` | Validates Claude's JSON response against the generated schema and retries up to N times if validation fails. On retry it augments the prompt with the specific validation error message. | Add to `.defaultAdvisors()` on the bean when extraction reliability in evaluation (Section 5) proves insufficient. Do not pre-emptively add — each retry is a full API call. Cap at `maxRepeatAttempts(2)`: three total calls (original + 2 retries) is the practical ceiling before the cost-per-document becomes unacceptable. |
| `@CircuitBreaker(name = "claude-api")` | Resilience4j decorator — opens the circuit after repeated Claude API failures, invoking the fallback method and returning null. | Must annotate a `public` method on a Spring-managed bean. Resilience4j uses CGLIB proxies and silently bypasses the circuit breaker on `private`, `protected`, or `final` methods with no startup error. |

### 3.5 Known Pitfalls

**Pitfall 1: Shared `ChatClient.Builder` state mutation**
Spring Boot auto-configures a single `ChatClient.Builder` bean as a singleton. Calling `.defaultSystem()` on it mutates the shared instance — whichever bean configures its system prompt last wins at runtime. Both beans end up with the same system prompt. Fix: always call `ChatClient.builder(chatModel)` (the static factory method, not the injected singleton builder) inside each `@Bean` method. This is already enforced in `AiClientConfig` (comment CR-05).

**Pitfall 2: Missing `@JsonProperty(required = true)` causes silent null fields**
Spring AI generates the JSON schema from your Java record via Jackson's schema generator. Without `required = true`, fields are marked optional in the generated schema. Claude may return `null` for them without Spring AI raising any error — the record deserializes successfully with null fields. The `NullPointerException` appears later when `StepExtractionService` tries to call `CareEventType.valueOf(step.eventType())` on a null string. Fix: annotate every field Claude must populate with `@JsonProperty(required = true, value = "fieldName")`.

**Pitfall 3: `@CircuitBreaker` on non-public methods is silently ignored**
Resilience4j's CGLIB proxies cannot intercept `private`, `protected`, or `final` methods. The annotation is ignored with no startup warning. The method runs without circuit breaker protection — a sustained Claude API outage will not open the circuit, and requests will pile up until the downstream timeout exhausts connection pool threads. Fix: every `@CircuitBreaker`-annotated method must be `public` on a Spring-managed bean (not `new`-instantiated).

**Pitfall 4: Unbounded `maxTokens` causes unpredictable latency and cost**
`AnthropicChatOptions.builder()` without `.maxTokens(...)` sends no token limit to the Anthropic API. If the system prompt is ambiguous or Claude decides to reason through the extraction in prose before outputting JSON, it may generate 3,000–5,000 tokens of explanation. For an extraction call where the output is a bounded JSON object, 2,000 tokens is a comfortable ceiling. Fix: always set `.maxTokens(2000)` on the `stepExtractionClient` bean.

**Pitfall 5: `CareEventType` enum deserialization fails hard on unknown values**
If Claude returns `"eventType": "TUMOR_BOARD_REVIEW"` and you declare the field as `CareEventType eventType` in the record, Jackson will throw `InvalidDefinitionException` during `.entity()`. This exception propagates out of the `try/catch` in `StepExtractionService` as an unlogged runtime error. Fix: declare `eventType` as `String` in the extraction record. Validate and map to `CareEventType` in service-layer code with an explicit try/catch and filter, not during deserialization. See `StepExtractionService.isValidCareEventType()` in Section 4.3.

## 4. Implementation Guidance

### 4.1 Model Configuration

**Model:** `claude-sonnet-4-20250514`
**Temperature:** `0.1` — extraction is a deterministic task; low temperature reduces hallucinated steps and encourages strict adherence to the enum constraint list
**Max tokens:** `2000` — sufficient for a JSON object containing 5–15 proposed steps with edge arrays; eliminates cost surprises from unbounded generation

Add the `stepExtractionClient` bean to `AiClientConfig.java` alongside the existing classification and alert beans:

```java
// AiClientConfig.java — add this @Bean
@Bean
ChatClient stepExtractionClient(ChatModel chatModel) {
    // ChatClient.builder(chatModel) — static factory creates a fresh builder.
    // Do NOT use the auto-configured ChatClient.Builder singleton here (CR-05).
    return ChatClient.builder(chatModel)
            .defaultSystem(ExtractionPrompts.SYSTEM_PROMPT)
            .defaultOptions(AnthropicChatOptions.builder()
                    .temperature(0.1)    // Deterministic extraction
                    .maxTokens(2000)     // Hard cap — bounded JSON output
                    .build())
            .build();
}
```

### 4.2 Output Schema — Java Records

`ExtractionResult` is a top-level wrapper record containing the proposed steps list and the detected duplicates list. `eventType` is declared as `String`, not `CareEventType`, to prevent hard Jackson deserialization failures on unknown values (Pitfall 5). Service-layer code validates and maps to the enum after deserialization.

```java
// com/onconavigator/ai/model/ExtractionResult.java
package com.onconavigator.ai.model;

import com.fasterxml.jackson.annotation.JsonProperty;
import com.fasterxml.jackson.annotation.JsonPropertyOrder;
import java.util.List;

/**
 * Top-level structured output record for Claude step extraction.
 *
 * <p>Spring AI generates a JSON schema from this record and instructs Claude to respond
 * with matching JSON. Fields marked required=true must always be present in Claude's output.
 *
 * <p>HIPAA note: this record contains only non-PHI care event metadata — event type codes,
 * step names as stated in the document, estimated timing, and extraction rationale text.
 * It does NOT contain patient identifiers. Never log instances of this record — rationale
 * text may re-state clinical content from PHI-bearing documents.
 */
@JsonPropertyOrder({"proposedSteps", "alreadyCoveredEventTypes"})
public record ExtractionResult(

    @JsonProperty(required = true, value = "proposedSteps")
    List<ProposedStep> proposedSteps,

    // Event types Claude identified in the document but skipped because they already
    // exist as ACTIVE/COMPLETED steps. Required even when empty — enables the
    // "already covered" transparency display (D-10). Claude must always return this list.
    @JsonProperty(required = true, value = "alreadyCoveredEventTypes")
    List<String> alreadyCoveredEventTypes
) {

    /**
     * A single proposed care pathway step extracted from the document.
     */
    @JsonPropertyOrder({"stepName", "eventType", "estimatedTimeWindowDays",
                         "proposedEdges", "extractionRationale"})
    public record ProposedStep(

        // Human-readable step name as it should appear in the pathway view
        @JsonProperty(required = true, value = "stepName")
        String stepName,

        // String, NOT CareEventType enum — validated and mapped in StepExtractionService.
        // Must be one of: REFERRAL, CONSULTATION, BIOPSY, PATHOLOGY_REPORT, IMAGING,
        // SURGERY, CHEMOTHERAPY, RADIATION, FOLLOW_UP, LAB_WORK, GENETIC_TESTING, OTHER
        @JsonProperty(required = true, value = "eventType")
        String eventType,

        // Estimated days from now or from a predecessor step. Null when not stated in document.
        @JsonProperty(value = "estimatedTimeWindowDays")
        Integer estimatedTimeWindowDays,

        // Proposed DAG edges for this step. Required, may be empty list.
        @JsonProperty(required = true, value = "proposedEdges")
        List<ProposedEdge> proposedEdges,

        // Quote or close paraphrase of the document sentence that motivated this extraction.
        // Shown to the nurse during inline step review for source traceability.
        @JsonProperty(required = true, value = "extractionRationale")
        String extractionRationale
    ) {}

    /**
     * A proposed dependency edge: predecessorStep must complete before this step begins.
     */
    @JsonPropertyOrder({"predecessorStepName", "predecessorIsExistingStep"})
    public record ProposedEdge(

        // Step name of the predecessor.
        // For existing steps: must match an existing step name exactly.
        // For proposed steps: must match a stepName in the same proposedSteps list.
        @JsonProperty(required = true, value = "predecessorStepName")
        String predecessorStepName,

        // True = predecessor is an already-active step in the patient's pathway.
        // False = predecessor is another step proposed in this same extraction.
        @JsonProperty(required = true, value = "predecessorIsExistingStep")
        boolean predecessorIsExistingStep
    ) {}
}
```

### 4.3 Core Extraction Service

`StepExtractionService` follows `DocumentClassificationService` exactly: `@CircuitBreaker` on a `public` method, feature flag via `@Value`, `null` return on failure, token budget truncation. The additional responsibility is `CareEventType` validation before returning the result.

```java
// com/onconavigator/ai/service/StepExtractionService.java
package com.onconavigator.ai.service;

import com.onconavigator.ai.prompt.ExtractionPrompts;

import io.github.resilience4j.circuitbreaker.annotation.CircuitBreaker;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.UUID;

/**
 * Claude-powered care step extraction from clinical documents.
 *
 * <p>Reads extracted document text and the patient's current pathway step context, then
 * asks Claude to identify care events mentioned in the document that are not yet tracked.
 * Returns a structured {@link ExtractionResult} with proposed steps and detected duplicates.
 *
 * <p>Gated by {@code onconavigator.ai.step-extraction.enabled} (default: false).
 * Must remain false until an Anthropic BAA is signed — document text contains PHI (D-13).
 *
 * <p>HIPAA note: document text (PHI) is sent to Claude. Never log document text, extracted
 * step names, or extractionRationale strings. Log only document UUIDs and counts.
 */
@Service
public class StepExtractionService {

    private static final Logger log = LoggerFactory.getLogger(StepExtractionService.class);

    // Matches DocumentClassificationService — documents are already capped at upload.
    // Re-truncate defensively in case the service is called outside the upload pipeline.
    private static final int MAX_INPUT_TOKENS = 150_000;
    private static final int CHARS_PER_TOKEN_ESTIMATE = 4;

    private final ChatClient stepExtractionClient;
    private final boolean extractionEnabled;

    public StepExtractionService(
            @Qualifier("stepExtractionClient") ChatClient stepExtractionClient,
            @Value("${onconavigator.ai.step-extraction.enabled:false}") boolean extractionEnabled) {
        this.stepExtractionClient = stepExtractionClient;
        this.extractionEnabled = extractionEnabled;
    }

    /**
     * Extract proposed pathway steps from clinical document text.
     *
     * <p>Returns null when:
     * <ul>
     *   <li>Feature flag disabled (BAA not in place)</li>
     *   <li>Extracted text is null or blank (document had no readable content)</li>
     *   <li>Circuit breaker is open (sustained Claude API outage)</li>
     *   <li>Claude returns unparseable or null JSON</li>
     * </ul>
     *
     * @param documentId          UUID of the source document (logging only — never log content)
     * @param extractedText       full document text (may contain PHI — never log)
     * @param existingStepsContext JSON summary of existing ACTIVE/COMPLETED steps (non-PHI:
     *                             step names and event type codes only — no patient identifiers)
     * @return extraction result with valid steps only, or null on any failure
     */
    @CircuitBreaker(name = "claude-api", fallbackMethod = "extractFallback")
    public ExtractionResult extractSteps(UUID documentId, String extractedText,
                                          String existingStepsContext) {
        if (!extractionEnabled) {
            log.info("Step extraction disabled (BAA not in place) for document {}", documentId);
            return null;
        }
        if (extractedText == null || extractedText.isBlank()) {
            log.info("Step extraction skipped — no extractable text in document {}", documentId);
            return null;
        }

        try {
            String truncated = truncateToTokenBudget(extractedText, documentId);

            ExtractionResult result = stepExtractionClient.prompt()
                    .user(u -> u.text(ExtractionPrompts.USER_TEMPLATE)
                                 .param("documentText", truncated)
                                 .param("existingSteps", existingStepsContext))
                    .call()
                    .entity(ExtractionResult.class);

            if (result == null || result.proposedSteps() == null) {
                log.warn("Step extraction returned null result for document {}", documentId);
                return null;
            }

            // Validate eventType strings before returning — prevents invalid enum values
            // from reaching PatientPathwayService and causing database constraint violations.
            // Invalid types are logged (value only, not step name) and filtered out.
            List<ExtractionResult.ProposedStep> validSteps = result.proposedSteps().stream()
                    .filter(step -> isValidCareEventType(step.eventType(), documentId))
                    .toList();

            int filtered = result.proposedSteps().size() - validSteps.size();
            if (filtered > 0) {
                log.warn("Step extraction: {} invalid CareEventType value(s) filtered for document {}",
                        filtered, documentId);
            }

            log.info("Step extraction completed for document {}: {} proposed, {} already covered, {} filtered",
                    documentId,
                    validSteps.size(),
                    result.alreadyCoveredEventTypes().size(),
                    filtered);

            return new ExtractionResult(validSteps, result.alreadyCoveredEventTypes());

        } catch (Exception e) {
            // Log message only — never log extractedText or result content
            log.error("Step extraction failed for document {}: {}", documentId, e.getMessage());
            return null;
        }
    }

    /**
     * Circuit breaker fallback — must be public for Resilience4j CGLIB proxy.
     *
     * <p>Called when the circuit breaker is open (sustained Claude API failures).
     * Returns null, triggering silent skip in DocumentProcessingService.
     */
    public ExtractionResult extractFallback(UUID documentId, String extractedText,
                                              String existingStepsContext, Exception e) {
        log.warn("Step extraction circuit breaker open for document {}: {}", documentId, e.getMessage());
        return null;
    }

    /**
     * Validate that an eventType string maps to a known CareEventType enum value.
     * Logs only the invalid value — not the step name (which may contain clinical content).
     */
    private boolean isValidCareEventType(String eventType, UUID documentId) {
        if (eventType == null) {
            log.warn("Step extraction returned null eventType for a step in document {}", documentId);
            return false;
        }
        try {
            CareEventType.valueOf(eventType);
            return true;
        } catch (IllegalArgumentException ex) {
            log.warn("Step extraction returned unknown CareEventType '{}' for document {}",
                    eventType, documentId);
            return false;
        }
    }

    private String truncateToTokenBudget(String text, UUID documentId) {
        int maxChars = MAX_INPUT_TOKENS * CHARS_PER_TOKEN_ESTIMATE;
        if (text.length() > maxChars) {
            log.warn("Document {} text truncated from {} to {} chars for token budget",
                    documentId, text.length(), maxChars);
            return text.substring(0, maxChars);
        }
        return text;
    }
}
```

The call is fire-and-forget from the upload response's perspective — the upload succeeds whether or not extraction succeeds. The actual Claude call should run on an `@Async` thread to avoid holding the database connection open during the API wait (see Section 4b.2):

### 4.6 State Management

Phase 6 has no conversational state. Each extraction is a stateless single-turn call:

- `StepExtractionService` holds no mutable state between calls
- `stepExtractionClient` is a stateless Spring singleton shared safely across all virtual threads
- Existing pathway steps are fetched fresh per-call from the database inside the async method
- Proposed steps are persisted to `PatientPathwayStep` with `status=PROPOSED`, `source=AI_EXTRACTED`, and a foreign key to `ClinicalDocument`

Java 21 virtual threads (enabled via `spring.threads.virtual.enabled=true`) handle concurrency at the HTTP layer. Blocking the virtual thread during the Claude API call (2–8 seconds) is acceptable — virtual threads yield the carrier thread while waiting on I/O.

Claude claude-sonnet-4-20250514 has a 200K token context window. The extraction call uses:

**Prompt caching:** The system prompt is stable across all extraction calls for all patients. ## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns for AI systems — adapted for Java/Spring Boot, Spring AI 1.1.0, and the Extraction system type.

### 4b.1 Structured Outputs with Java Records + BeanOutputConverter

**The core rule:** Claude's response is text. Never treat it as guaranteed-valid JSON and deserialize directly. Define your schema as a Java record with `@JsonProperty` annotations, let Spring AI generate the JSON schema and inject format instructions, and validate the result before using it.

**How Spring AI's `.entity()` works under the hood:**

`ChatClient.entity(ExtractionResult.class)` triggers Spring AI's `BeanOutputConverter` internally:
1. Generates a JSON schema from `ExtractionResult` via Jackson's schema generator
2. Appends format instructions to the user prompt: "Respond with JSON conforming to this schema: {...}"
3. Calls Claude
4. Calls `Jackson.readValue(response, ExtractionResult.class)`
5. Returns the typed record or throws if Jackson deserialization fails

The `@JsonProperty(required = true)` annotation propagates to the generated schema, making Claude aware that the field is mandatory. Without it, the schema marks the field as optional and Claude may omit it with no error.

**`eventType` as `String`, not enum — the deliberate choice:**

If you declare `CareEventType eventType` in the record, Jackson throws `InvalidDefinitionException` when Claude returns an unknown value like `"TUMOR_BOARD_REVIEW"`. This exception is unchecked and propagates up the call stack if not caught at the right layer. Declaring `String eventType` and validating in service code gives you control over the failure mode: you can filter the step out, log the unknown value, and return a valid (shorter) result instead of null for the whole extraction.

**Native structured output as an alternative (verify before enabling):**

Adding `.advisors(AdvisorParams.ENABLE_NATIVE_STRUCTURED_OUTPUT)` sends the JSON schema as a structured output constraint directly in the Anthropic API request rather than embedding format instructions in the prompt text. The model is constrained at the API level — it cannot output non-conforming JSON. This eliminates the format instruction token overhead and improves reliability. Verify that `spring-ai-anthropic:1.1.0` supports this advisor for Anthropic models before enabling in production (the feature is confirmed for OpenAI and Mistral; Anthropic support should be verified against the changelog).

**`StructuredOutputValidationAdvisor` — the retry mechanism:**

Add this when evaluation (Section 5) shows extraction JSON is malformed on more than ~5% of calls:

```java
// AiClientConfig.java — add validationAdvisor to stepExtractionClient when needed
var validationAdvisor = StructuredOutputValidationAdvisor.builder()
        .outputType(ExtractionResult.class)
        .maxRepeatAttempts(2)   // Up to 3 total calls; set no higher
        .advisorOrder(BaseAdvisor.HIGHEST_PRECEDENCE + 1000)
        .build();

return ChatClient.builder(chatModel)
        .defaultSystem(ExtractionPrompts.SYSTEM_PROMPT)
        .defaultAdvisors(validationAdvisor)
        .defaultOptions(AnthropicChatOptions.builder()
                .temperature(0.1)
                .maxTokens(2000)
                .build())
        .build();
```

On retry, Spring AI augments the prompt with the specific Jackson error message from the previous attempt, helping Claude correct the malformed field.

**What to log on failure:**
- Log: document UUID, exception class name (`e.getClass().getSimpleName()`), attempt number
- Never log: Claude's raw response string (may contain re-stated PHI from the document), `extractionRationale` field values, step name strings

**When to surface to the user:**
- Extraction failure: silent skip. The upload succeeds, no proposed steps appear, and the nurse proceeds with manual pathway entry as before. Extraction is enhancement, not a prerequisite.
- Circuit breaker open: log at WARN. Surface as a system health alert to the ADMIN role via Spring Actuator's audit events endpoint. Never surface Claude API failures in the nurse-facing UI.

### 4b.2 Virtual Threads + @Async (Java/Spring Context)

**How concurrency works in this stack:**

Spring Boot 3.5 with `spring.threads.virtual.enabled=true` runs each incoming HTTP request on a Java 21 virtual thread. The Claude API call inside `StepExtractionService.extractSteps()` is a blocking HTTP call — it blocks the calling thread for 2–8 seconds. With virtual threads this is acceptable: blocking a virtual thread yields the carrier thread (a platform thread from the ForkJoinPool) while waiting on I/O. The application can handle thousands of concurrent uploads without exhausting platform threads.

**The one mistake to avoid — holding a `@Transactional` database connection across the Claude call:**

`DocumentProcessingService.processUpload()` is annotated `@Transactional`. The database transaction holds an open connection from the HikariCP pool for the entire method duration. If `extractSteps()` is called inline inside `processUpload()`, the connection is held for 2–8 seconds while waiting for the Claude API. Under modest load (10 concurrent uploads), all HikariCP connections are exhausted and subsequent requests queue or time out.

Fix: run extraction after the transaction commits, on a separate thread via `@Async`. The upload response returns before Claude is called:

```java
// StepExtractionTriggerService.java — a thin @Async wrapper
@Service
public class StepExtractionTriggerService {

    private final StepExtractionService extractionService;
    private final PatientPathwayService pathwayService;

    // Constructor injection (omitted for brevity)

    /**
     * Trigger step extraction asynchronously after the upload transaction commits.
     * The @Async annotation causes Spring to execute this method on a separate
     * virtual thread, decoupled from the caller's transaction.
     *
     * @param documentId   source document UUID
     * @param patientId    linked patient UUID
     * @param extractedText document text (PHI — never log)
     */
    @Async
    @Transactional
    public void triggerAsync(UUID documentId, UUID patientId, String extractedText) {
        String existingStepsContext = pathwayService.buildExistingStepsContext(patientId);
        ExtractionResult result =
                extractionService.extractSteps(documentId, extractedText, existingStepsContext);
        if (result != null) {
            pathwayService.createProposedSteps(patientId, documentId, result);
            // Fire pathwayStepsChanged Temporal signal (Phase 5 D-06)
            pathwayService.signalPathwayStepsChanged(patientId);
        }
    }
}
```

**`@EnableAsync` requirement:** `@Async` requires `@EnableAsync` on a `@Configuration` class. Spring Boot does not enable it automatically. Add `@EnableAsync` to `AiClientConfig` or a dedicated `AsyncConfig` class — without it, `@Async` methods run synchronously on the caller's thread with no error or warning.

**Stream vs. blocking call for extraction:**

Do not use `chatClient.stream()` for structured extraction. Streaming delivers response tokens incrementally — the JSON object is only parseable once all tokens arrive. Spring AI's `BeanOutputConverter` cannot deserialize a partial JSON stream. Streaming adds complexity and an incremental parser with no user-facing benefit for a background job. Use `.call().entity()` (synchronous, blocking the virtual thread).

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt — why separation matters here:**

The system prompt (`ExtractionPrompts.SYSTEM_PROMPT`) establishes the AI persona, the valid enum value list, all output rules, and the negation constraint. It is injected at bean construction time via `.defaultSystem()` and is identical for every document extraction call. This stability makes it eligible for Anthropic prompt caching — after the first call it is billed at 10% of normal input token cost.

The user template (`ExtractionPrompts.USER_TEMPLATE`) receives the per-call variables: `{documentText}` and `{existingSteps}`. Putting the enum list or output rules in the user template would disqualify the static portion from caching and bloat every API request unnecessarily.

**Enum constraint injection — keep it in the system prompt and keep it in sync:**

The system prompt enumerates every valid `CareEventType` value. Claude does not introspect Java enums at runtime — it only knows what the prompt tells it. When a new value is added to `CareEventType`, update `ExtractionPrompts.SYSTEM_PROMPT` in the same commit. If the enum grows beyond 15–20 values, consider generating the list programmatically:

```java
// ExtractionPrompts.java — dynamic enum injection to stay in sync automatically
public static String buildSystemPrompt() {
    String validTypes = Arrays.stream(CareEventType.values())
            .map(Enum::name)
            .collect(Collectors.joining(", "));
    return SYSTEM_PROMPT_TEMPLATE.replace("{validEventTypes}", validTypes);
}
```

Call `buildSystemPrompt()` in `AiClientConfig` instead of referencing the constant directly.

**Few-shot examples:**

Inline static examples are appropriate for extraction. The output format is bounded, the examples are short (2–3 are sufficient), and the document corpus is uniform enough that a single set of examples covers the common patterns: an explicit treatment order, an inferred sequence, and a negated item that should NOT be extracted. Dynamic example retrieval via embeddings adds infrastructure complexity that is not justified for this use case at pilot scale.

**`maxTokens` is non-negotiable in production:**

The `stepExtractionClient` bean sets `maxTokens(2000)`. Never deploy without this. A document with ambiguous structure can cause Claude to reason through the extraction in prose before outputting JSON, burning 3,000+ tokens and triggering a 30-second response that will time out Tomcat's request thread before the response arrives.

### 4b.4 Context Window Management for Extraction

**The full context budget per call:**

| Component | Budget | Notes |
|---|---|---|
| System prompt | ~250 tokens | Stable, cached after first call |
| Existing steps context | ~100–500 tokens | Non-PHI step names + event types |
| Spring AI format instructions | ~200–400 tokens | Injected by `BeanOutputConverter` |
| Document text | up to 150,000 tokens | Hard-capped by `truncateToTokenBudget()` |
| Output | up to 2,000 tokens | Hard-capped by `maxTokens` |
| **Headroom** | ~47,000 tokens | Buffer for future prompt additions |

**Document truncation trade-offs:**

Simple tail-truncation (remove the last N characters) drops the end of long documents. For multi-page discharge summaries or operative notes, the treatment plan lives in the "Assessment and Plan" section, which in most EMR templates appears in the second half of the document. Tail-truncation at 150K tokens (600,000 characters) will capture this section for any document under ~100 typed pages. Truncation is a risk only for unusually long documents.

If evaluation (Section 5) finds that a specific document type consistently misses steps due to truncation, the preprocessing step in `DocumentProcessingService.extractText()` can be extended to reorder the document — extract and front-load lines containing section headers like "PLAN:", "ORDERS:", "ASSESSMENT AND PLAN:" before sending to Claude.

**Existing steps context size management:**

A typical oncology patient has 5–15 pathway steps. At ~20–30 tokens per step entry, the context is 100–450 tokens — well within budget. For a hypothetical patient with 100+ steps (multi-year complex case), truncate to the 50 most recent steps (most relevant for detecting duplicates). The full step history does not need to be in the prompt — Claude only needs to know what is currently ACTIVE or COMPLETED.

**What never goes in the existing steps context:**

No patient identifiers (MRN, name, DOB). The context is non-PHI:
```json
[{"stepName": "Initial Consultation", "eventType": "CONSULTATION", "status": "COMPLETED"},
 {"stepName": "Core Needle Biopsy", "eventType": "BIOPSY", "status": "ACTIVE"}]
```
Patient identity is already established from the document text. Including it in the steps context would add PHI exposure with no extraction benefit and would violate the minimum-necessary principle.

| Input: existing steps + format instructions | ~600 | ~$0.002 |

At a V1 pilot practice uploading 20–50 documents per day: $0.40–$9.00/day, $12–$270/month. This is within budget for a single-practice pilot.

**Latency budget:**

Claude claude-sonnet-4-20250514 extraction calls return in approximately 3–10 seconds for typical oncology documents (2,000–20,000 input tokens). The `@Async` pattern (Section 4b.2) decouples this latency from the upload response — the nurse sees upload confirmation in under 500ms. Proposed steps appear in the pathway view seconds later via the existing TanStack Query polling or `pathwayStepsChanged` signal refresh.

**Prompt caching impact on cost:**

With `AnthropicCacheStrategy.SYSTEM_ONLY`, the system prompt (~250 tokens) is cached after the first call and billed at 10% of the normal input rate thereafter. At 50 documents/day, this saves approximately $0.03/day — trivial at pilot scale. Configure it correctly now so the saving scales automatically as volume grows.

**Cheaper models for sub-tasks:**

The `CareEventType` string validation and duplicate detection are deterministic Java logic — no LLM needed for either. If a pre-extraction relevance filter is added in the future ("is this document an oncology clinical note at all?"), use `claude-haiku-3-5` for that binary classification call. Never use `claude-sonnet-4` for a classification that precedes the extraction — the cost ratio is approximately 10:1 and the classification task does not require Sonnet's reasoning depth.

**Application-layer caching:**

Exact-match caching of extraction results by document text hash is not useful — no two clinical documents are identical. However, the `existingStepsContext` string for a given patient is stable across multiple document uploads in a short window. Cache it per patient UUID with a 60-second TTL using Spring's `@Cacheable` on `buildExistingStepsContext(UUID patientId)`. This avoids redundant database queries when a nurse uploads several documents for the same patient in quick succession.

### 5.1 Tooling Decision

No existing eval tooling was detected in this codebase (no Langfuse, LangSmith, Arize Phoenix, Braintrust, or Promptfoo dependencies). The project is a pure Java/Spring Boot application with no Python runtime.

**Selected approach: JUnit-based corpus evaluation against the existing test-corpus/**

This keeps eval tests out of the default `mvn test` cycle (which should remain fast and offline).

### 5.2 Evaluation Dimensions

The five dimensions below map directly from the domain rubric ingredients in Section 1b. They are ordered by priority. Each includes a concrete PASS/FAIL rubric in domain language and the measurement approach.

---

#### Dimension 1: Document Fidelity (No Hallucinated Steps)

**Priority: Critical**

Every proposed step must be traceable to a specific sentence or phrase in the source document. A step is a hallucination if it represents standard-of-care knowledge that was not explicitly stated in this document.

**PASS:** The referral letter `referral-medical-oncology-01.txt` discusses concurrent chemoradiation followed by durvalumab consolidation and coordination with radiation oncology. The extraction proposes steps for CHEMOTHERAPY, RADIATION, and CONSULTATION (radiation oncology). No step for "CT follow-up at 3 months" or "bone density scan" appears — those are plausible for this patient's cancer type but absent from this document.

**FAIL:** The extraction proposes a FOLLOW_UP step for "Cardiology clearance" based on the patient's documented coronary artery disease history — a clinically reasonable but document-absent extrapolation from the patient's comorbidities section.

**Measurement: Human review** — a nurse navigator reviews each proposed step against the source document text and marks it as "document-supported" or "not in document." This is the ground-truth label for the reference dataset.

**Metric:** Precision = document-supported steps / total proposed steps. Target >= 0.95 before pilot deployment. A hallucination rate above 5% means the nurse spends more time rejecting AI proposals than confirming them, negating the value proposition.

---

#### Dimension 2: Completeness (No Missed Treatment Plan Items)

**Priority: High**

All explicitly stated or clearly ordered care events in the document must appear as proposed steps. A missed step is a monitoring blind spot — the pathway engine cannot alert on something it does not know to watch.

**PASS:** The combined report `multi-section-path-rad-01.txt` contains six distinct tumor board recommendations (targeted ultrasound, possible biopsy of second focus, conditional mastectomy or lumpectomy, sentinel lymph node biopsy, Oncotype DX genomic testing, contralateral breast MRI surveillance). The extraction proposes at least five steps covering these items. The conditional "if second focus is malignant" branches may reasonably be compressed to one step.

**FAIL:** The extraction from `multi-section-path-rad-01.txt` returns only two steps (IMAGING and SURGERY), missing the BIOPSY of the second focus, GENETIC_TESTING (Oncotype DX), and FOLLOW_UP (MRI surveillance). The nurse, relying on AI pre-population, does not notice the omissions and the pathway misses three monitoring milestones.

**Measurement: Human review + code assist** — the nurse navigator labels the reference dataset with the expected step count and event types per document. The JUnit eval test asserts that the extracted step count is within 1 of the expected count and that all expected event types appear in the extracted set.

**Metric:** Recall = extracted expected event types / total expected event types in label. Target >= 0.85 across the reference dataset (acceptable that some abbreviation-heavy or parenthetical items may be missed; unacceptable to miss items stated explicitly in numbered lists or section headers).

---

#### Dimension 3: Clinical Term Normalization (Valid CareEventType Mapping)

**Priority: Critical**

Every extracted step must have an `eventType` that maps to a valid `CareEventType` enum value. An invalid value is filtered by `StepExtractionService.isValidCareEventType()` and never reaches the database — but a high filter rate means steps are silently dropped, producing the same outcome as missed steps.

**PASS:** "EBUS-TBNA" (endobronchial ultrasound transbronchial needle aspiration) in `referral-medical-oncology-01.txt` is mapped to BIOPSY. "Durvalumab consolidation" is mapped to CHEMOTHERAPY. "Radiation oncology coordination" is mapped to CONSULTATION or REFERRAL. All returned `eventType` strings are in the `CareEventType` enum.

**FAIL:** Claude returns `"eventType": "EBUS_BIOPSY"` or `"eventType": "IMMUNOTHERAPY"` — values that are clinically meaningful but absent from the enum. `isValidCareEventType()` filters them. The nurse sees fewer proposed steps than the document warrants, with no explanation.

**Measurement: Code-based** — the JUnit eval test calls `CareEventType.valueOf()` on every `eventType` string in the extraction result before any filtering is applied. Assert that invalid-type count is 0 across all reference dataset documents. This is a hard gate: any invalid enum value from Claude represents a prompt or schema instruction failure.

```java
// StepExtractionEvalTest.java — assert zero invalid event types
ExtractionResult rawResult = captureRawResultBeforeFiltering(documentText);
long invalidCount = rawResult.proposedSteps().stream()
    .filter(s -> !isValidCareEventType(s.eventType()))
    .count();
assertThat(invalidCount)
    .as("Document %s produced %d invalid CareEventType values", documentId, invalidCount)
    .isZero();
```

**Metric:** Invalid event type rate = 0 per reference dataset run. If non-zero, update `ExtractionPrompts.SYSTEM_PROMPT` to clarify the failing mapping before re-running.

---

#### Dimension 4: Negation Correctness

**Priority: Critical**

Negated, contraindicated, or explicitly ruled-out items must not appear as proposed steps. This is a direct trust-breaking failure: a nurse who sees "Radiation Therapy — PROPOSED" after reading "radiation is contraindicated" loses confidence in all AI proposals.

**PASS:** A document stating "Patient is not a surgical candidate given ECOG performance status 3" produces zero SURGERY steps. A document stating "radiation is contraindicated due to prior XRT" produces zero RADIATION steps.

**FAIL:** The extraction proposes SURGERY with `extractionRationale: "Patient is not a surgical candidate"` — Claude extracted the topic of the sentence rather than interpreting the negation.

**Measurement: Code-based + human review** — the reference dataset must include at least 3 documents with explicitly negated items (add these to `test-corpus/edge-cases/` if not already present: a note with surgical contraindication, a note with radiation contraindication, a note with a drug allergy that might be misread as a medication step). The JUnit eval test asserts that negated event types do not appear in the extraction result.

**Metric:** Negation error rate = 0 on labeled negation test cases. Even one negation failure in the reference dataset is a prompt-quality issue that must be resolved before pilot use.

---

#### Dimension 5: Noise Rejection (Administrative Content Excluded)

**Priority: Medium**

Documents from the practice contain administrative boilerplate (fax numbers, insurance pre-authorization instructions, scheduling logistics, cc: lists). These must not become proposed pathway steps.

**PASS:** `referral-medical-oncology-01.txt` contains the referring physician's fax number, office address, and a closing "please coordinate with radiation oncology" sentence. The extraction proposes CONSULTATION (radiation oncology) as a clinical step but does not propose "Fax results to Dr. SampleDoc" or "Obtain insurance pre-authorization" as steps.

**FAIL:** The extraction proposes a step with `stepName: "Fax pathology results to referring physician"` and `eventType: OTHER` because the referral letter contained "please fax results to (555) 555-0202."

**Measurement: Human review** — the nurse navigator reviews proposed steps on the reference dataset documents and flags any step that is administrative rather than clinical. The JUnit eval test counts flagged-administrative steps per document.

**Metric:** Administrative noise rate <= 1 step per 20 proposed steps across the reference dataset. A low rate is acceptable at pilot scale because the nurse review step catches it; a high rate triggers a prompt revision to strengthen the "do not extract administrative actions" instruction.

---

#### Dimension 6: Sequencing and Dependency Accuracy

**Priority: High**

Proposed edges must reflect ordering stated or strongly implied in the document. Edges inferred from general oncology convention rather than document content override physician intent (PW-ALL-002). DAG cycles corrupt the pathway graph structurally.

**PASS:** The operative note `operative-lumpectomy-01.txt` states specimens were sent to pathology. The extraction proposes SURGERY with a proposedEdge to PATHOLOGY_REPORT (predecessorIsExistingStep: false), reflecting that pathology follows surgery. No edge from PATHOLOGY_REPORT back to SURGERY is proposed. A biopsy note that mentions two independent tests with no stated sequencing proposes both steps with empty proposedEdges lists.

**FAIL:** The extraction from a lung cancer referral proposes edges BIOPSY -> IMAGING -> CHEMOTHERAPY inferring standard staging-then-treatment sequence, when the document only states "the patient requires chemotherapy evaluation" with no sequencing language. The oncologist's intent may have been to start systemic therapy while imaging is still pending.

**Measurement: Code-based + human review** — the JUnit eval test runs a topological sort on the proposed edge graph for each extraction result and asserts no cycles are present (this is a hard structural invariant; a cycle would corrupt the DAG). Edge direction correctness requires human review: the oncologist labels whether each proposed edge in the reference dataset reflects document-stated sequencing or convention inference.

```java
// StepExtractionEvalTest.java — assert no DAG cycles in proposed edges
assertNoCycles(result.proposedSteps()); // topological sort, throw if cycle detected
```

**Metric:** DAG cycle rate = 0 (hard gate). Document-supported edge rate (edge direction matches document intent) >= 0.80 per human review of labeled reference dataset.

```xml
    </build>

```

The eval profile:
- Includes only `*EvalTest.java` files (not unit/integration tests)
- Sets the feature flag to `true` so `StepExtractionService` calls Claude rather than returning null
- Injects the API key from a CI secret (`ANTHROPIC_API_KEY` in GitHub Actions secrets)
- Uses a real `StepExtractionService` bean wired with the real `ChatClient` — not mocked

**In CI (GitHub Actions):** Run the eval profile on push to `main` and on pull requests that touch `ExtractionPrompts.java`, `StepExtractionService.java`, or `ExtractionResult.java`. This makes prompt regressions visible before merge.

```yaml
# .github/workflows/eval.yml — add as a separate workflow
- name: Run extraction eval suite
  if: |
    contains(github.event.commits.*.modified.*, 'ExtractionPrompts.java') ||
    contains(github.event.commits.*.modified.*, 'StepExtractionService.java')
```

**Gating policy:** The eval suite is informational during development — failures do not block merge. Before pilot go-live, promote the eval gate to blocking on the following thresholds:
- Negation error rate: 0 (any negation failure blocks)
- Invalid CareEventType rate: 0 (any invalid type blocks)
- DAG cycle rate: 0 (any cycle blocks)
- Precision (document fidelity): >= 0.90
- Recall (completeness): >= 0.80

## 6. Guardrails

Guardrails are classified as **online** (runs on every request, real-time, must be fast) or **offline flywheel** (sampled batch, feeds improvement loop). Each online guardrail adds latency to the async extraction path — keep them minimal and fast.

The human-in-the-loop gate (nurse confirmation before PROPOSED steps become ACTIVE) is the primary safety layer for this system. These guardrails operate before and after the Claude call, not as replacements for nurse review.

---

### 6.1 Online Guardrails (Runs on Every Extraction)

These execute inside `StepExtractionService.extractSteps()` synchronously on the async extraction thread. Total overhead target: < 5ms.

| Guardrail | What It Checks | Implementation | Failure Action |
|---|---|---|---|
| **Feature flag gate** | `onconavigator.ai.step-extraction.enabled` is false until Anthropic BAA is signed | `if (!extractionEnabled) return null` at method entry — already in `StepExtractionService` | Silent skip: upload succeeds, no proposed steps |
| **Blank document guard** | Document has extractable text before sending ePHI to Claude API | `if (extractedText == null || extractedText.isBlank()) return null` — already in `StepExtractionService` | Silent skip: saves an API call and avoids sending an empty prompt |
| **Token budget truncation** | Document text does not exceed 150K token estimate before sending | `truncateToTokenBudget()` — already in `StepExtractionService` | Truncate silently, log at WARN with document UUID and char counts (no PHI in log) |
| **CareEventType enum validation** | No invalid eventType strings reach the database | `isValidCareEventType()` filter on each step before returning — already in `StepExtractionService` | Filter the invalid step, log at WARN with the invalid value only (not step name), return reduced result |
| **DAG cycle detection** | Proposed edges do not form cycles in the patient's pathway graph | Topological sort on (existing steps + proposed steps + proposed edges) before calling `createProposedSteps()` — implement in `PatientPathwayService.validateProposedEdges()` | Remove the cycle-creating edges from the proposal, log at WARN, return the acyclic subset |
| **Null result guard** | Claude returned null or empty `proposedSteps` list | `if (result == null || result.proposedSteps() == null) return null` — already in `StepExtractionService` | Silent skip |

**PROPOSED status enforcement** is not a runtime guardrail — it is enforced structurally. `PatientPathwayService.createProposedSteps()` always creates steps with `status=PROPOSED` and `source=AI_EXTRACTED`. There is no code path from `StepExtractionService` to `status=ACTIVE`. The confirm/reject transition is only available through the explicit `confirmProposedStep()` and `rejectProposedStep()` endpoints, which are `@PreAuthorize`-protected for `ROLE_NURSE_NAVIGATOR`. Auto-activation is architecturally impossible, not just guarded.

---

### 6.2 Circuit Breaker (Infrastructure Guardrail)

The Resilience4j `@CircuitBreaker(name = "claude-api")` shared with `DocumentClassificationService` handles sustained Claude API failures. When the circuit opens:

- `extractFallback()` returns null — extraction silently skips
- The nurse proceeds with manual pathway entry (the system's fallback mode)
- A WARN log entry records the circuit breaker open event with document UUID
- Spring Actuator `/actuator/health` reflects the circuit breaker state — surfaced to `ROLE_ADMIN` in the system health panel

No circuit breaker configuration changes are needed for Phase 6 — the existing `claude-api` breaker shared with classification is the correct scope. A Claude API outage affects both features simultaneously; they should open and close together.

---

### 6.3 Offline Flywheel (Sampled Batch Evaluation)

These run outside the request path, on a schedule or triggered by admin action. They produce the quality signal that drives prompt improvement over time.

| Signal | What It Measures | Collection Method | Review Cadence |
|---|---|---|---|
| **Nurse rejection rate** | Percentage of PROPOSED steps that nurses reject vs. confirm | `PatientPathwayStep` table: `COUNT(status=REJECTED) / COUNT(status IN (ACTIVE, REJECTED))` grouped by `source=AI_EXTRACTED` and week | Weekly during pilot |
| **Invalid CareEventType rate** | Percentage of Claude-returned steps that failed enum validation | Log aggregation: count WARN messages matching "invalid CareEventType" per day | Weekly during pilot |
| **Nurse edit rate before confirm** | Percentage of confirmed steps where the nurse edited stepName or eventType before confirming | Track `original_step_name != confirmed_step_name` or `original_event_type != confirmed_event_type` in `PatientPathwayStep` audit fields | Weekly during pilot |
| **Extraction null rate** | Percentage of document uploads where `StepExtractionService` returned null (any cause) | Log aggregation: count INFO messages "Step extraction skipped/disabled" and ERROR messages per day | Weekly during pilot |
| **Per-document-type performance** | Whether rejection rate or invalid-type rate is higher for specific document types (operative notes vs. referrals vs. lab results) | Group rejection rate by `ClinicalDocument.documentType` | Monthly or on regression |

**Rejection rate threshold for prompt review:** If the week-over-week nurse rejection rate exceeds 20% (meaning nurses are rejecting more than 1 in 5 AI proposals), review a sample of 10 rejected steps against their source documents to identify the failure pattern before modifying the prompt. Do not modify the prompt based on rejection rate alone — a high rejection rate could indicate hallucination, miscategorization, or simply that the documents uploaded that week contained unusual content.

**Edit rate threshold:** If the nurse edit rate before confirm exceeds 30%, the most common edit type reveals where the prompt needs improvement: if nurses are mostly changing event types, the enum mapping instructions need strengthening; if they are mostly editing step names, the stepName generation quality needs attention.

## 7. Production Monitoring

### 7.1 Tracing and Observability

**Selected approach: Spring Boot Actuator custom metrics (Micrometer) — no new infrastructure required**

The project already has `spring-boot-starter-actuator` in the dependency stack. Micrometer counters and timers in `StepExtractionService` and `StepExtractionTriggerService` give the necessary visibility through the existing `/actuator/metrics` endpoint and any Prometheus/Grafana setup that may already be in place for the application.

This avoids adding Arize Phoenix, Langfuse, or any Python-based observability platform to a Java application at pilot scale. If the practice scales to multiple users and the team wants trace visualization, Arize Phoenix (open-source, self-hostable via Docker) can be added as a Docker Compose service using OpenTelemetry — the instrumentation points below are forward-compatible.

**Instrumentation to add in `StepExtractionService`:**

```java
// Inject MeterRegistry into StepExtractionService constructor
private final MeterRegistry meterRegistry;

// In extractSteps(), after the valid-steps filter:
meterRegistry.counter("extraction.steps.proposed",
    "document_type", documentType).increment(validSteps.size());
meterRegistry.counter("extraction.steps.filtered_invalid",
    "document_type", documentType).increment(filtered);
meterRegistry.timer("extraction.latency_ms",
    "document_type", documentType).record(elapsedMs, TimeUnit.MILLISECONDS);

// On null return paths:
meterRegistry.counter("extraction.result.null",
    "reason", reason).increment(); // reason: "feature_disabled", "blank_document", "api_error", "circuit_open"
```

**Available via `/actuator/metrics`:**
- `extraction.steps.proposed` — total steps proposed over time
- `extraction.steps.filtered_invalid` — invalid CareEventType count (should approach 0)
- `extraction.latency_ms` — p50/p95/p99 latency trend (alert if p95 > 15 seconds)
- `extraction.result.null` — null return rate by reason

---

### 7.2 Key Metrics and Alert Thresholds

| Metric | Collection | Alert Threshold | Action |
|---|---|---|---|
| **Invalid CareEventType rate** | `extraction.steps.filtered_invalid` / `extraction.steps.proposed` | > 5% over a rolling 24-hour window | Review recent extraction logs (invalid values only — no PHI) against the prompt's enum list; likely a new document type with unusual terminology |
| **Extraction null rate** | `extraction.result.null{reason="api_error"}` count | > 3 in any 10-minute window | Check Resilience4j circuit breaker state via `/actuator/health`; likely a Claude API outage; circuit breaker will handle recovery automatically |
| **Extraction latency p95** | `extraction.latency_ms` p95 | > 15 seconds | Review recent documents for unusually large inputs; check if token budget truncation is firing; Claude API may be degraded |
| **Nurse rejection rate** | Database query: rejected / (active + rejected) for AI-sourced steps | > 20% in a rolling 7-day window | Manually review 10 rejected steps against source documents; identify pattern before prompt change |
| **Circuit breaker state** | Resilience4j `/actuator/health` | State transitions to OPEN | Log at WARN; surface in admin health panel; no nurse-facing alert |

Alert delivery: log at ERROR level for threshold breaches. At V1 pilot scale, manual log review is sufficient. Add email notification via Spring Boot's `ApplicationEventPublisher` + an `@EventListener` for the circuit breaker open event if the practice wants proactive notification.

---

### 7.3 Sampling Strategy

At V1 pilot scale (20–50 documents/day), **100% of extractions are reviewable** — the nurse reviews every proposed step before confirming. There is no sampling needed for safety: the nurse IS the sample.

For the offline flywheel (Section 6.3), sample review cadence:

- **Weekly:** Review all extraction quality metrics (rejection rate, edit rate, invalid-type rate, null rate) from the database and Actuator metrics. This takes 15 minutes and catches regressions before they compound.
- **On threshold breach:** When any alert threshold is hit, pull the 10 most recent extraction events where the metric breached. Review the document UUID and extracted step counts in the logs (no PHI exposure). If the pattern is clear from metadata alone, address it without accessing document content. If the pattern requires document review, access through the application's authenticated UI under HIPAA audit logging.
- **Monthly:** Run the full eval suite against the reference dataset (`./mvnw test -Peval`) to confirm no prompt or model regression. Compare results to the pre-deployment baseline.
- **On prompt or model change:** Run the full eval suite before deploying any change to `ExtractionPrompts.java` or `application.yml` model configuration. Treat eval suite results as the pre-deployment gate.

---

### 7.4 HIPAA Logging Constraints for Monitoring

All monitoring data must comply with HIPAA minimum-necessary and PHI-in-logs constraints established in the project:

| Allowed in monitoring | Not allowed in monitoring |
|---|---|
| Document UUID | Document text content |
| Step count (integers) | Step names (may contain clinical content re-stated from the document) |
| Invalid CareEventType string value (e.g., "TUMOR_BOARD_REVIEW") | `extractionRationale` field values |
| Latency in milliseconds | Patient identifiers |
| Null return reason enum | Any field from `ExtractionResult.ProposedStep` |
| Circuit breaker state transitions | Claude's raw API response string |

Metrics exposed via `/actuator/metrics` contain only the allowed items above. The endpoint is restricted to `ROLE_ADMIN` via Spring Security `@PreAuthorize` configuration already established in Phase 1.
