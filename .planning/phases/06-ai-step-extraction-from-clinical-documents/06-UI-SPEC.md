## Spacing Scale

Declared values (multiples of 4 only). Inherited from Phase 5 — no changes for Phase 6.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge internal padding, inline separators |
| sm | 8px | Compact button padding, gap between confirm/reject buttons |
| md | 16px | Default element spacing, step row padding (p-3 = 12px, step list gap) |
| lg | 24px | Section padding, depth indent per level (24px × depth) |
| xl | 32px | Card content padding |
| 2xl | 48px | Major section breaks between cards |
| 3xl | 64px | Page-level vertical spacing |

Exceptions:
- Touch targets: minimum 44px height on all interactive rows and buttons. Step rows use `min-h-[44px]` — maintain this for all new proposed step rows and confirm/reject button rows.
- Confirm/Reject icon buttons: `h-8 w-8` (32px) — same as existing edit/remove buttons in StepRow. Acceptable because they are paired with a text label or tooltip.

Source: Existing `StepRow.tsx` (`p-3`, `min-h-[44px]`), `PathwayEditor.tsx` (`space-y-1`, `space-y-2`), established pattern.

## Typography

Inherited from existing StepRow and PathwayEditor patterns — Phase 6 introduces no new type roles.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (text-sm) | 400 (regular) | 1.5 (leading-normal) |
| Label | 14px (text-sm) | 600 (semibold) | 1.25 (leading-snug) |
| Caption / timing | 12px (text-xs) | 400 (regular) | 1.5 (leading-normal) |
| Section heading | 16px (text-base) | 600 (semibold) | 1.2 (leading-tight) |

Rules specific to Phase 6:
- PROPOSED step names: `text-sm text-muted-foreground` (weight 400, muted color) — same as existing PROPOSED styling in StepRow.
- ACTIVE/CONFIRMED step names: `text-sm font-semibold` — unchanged from existing pattern.
- "AI Proposed" badge label: `text-xs font-semibold` (weight 600 — visually distinct from surrounding muted body text, matches badge conventions).
- "Already covered" section header: `text-xs font-semibold text-muted-foreground uppercase tracking-wide`.
- "Already covered" item names: `text-sm text-muted-foreground`.

Source: `StepRow.tsx` `stepNameClass()` function, existing typography patterns.

## Color

All tokens are OKLCh CSS custom properties from `src/app.css`. Phase 6 uses only established tokens — no new color tokens are introduced.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` oklch(1 0 0) | Page background, step list background |
| Secondary (30%) | `--card` oklch(1 0 0) / `--secondary` oklch(0.97 0 0) | Cards containing pathway panel, "already covered" section background |
| Accent (10%) | `--primary` oklch(0.205 0 0) | Confirm button (primary action only) |
| Destructive | `--destructive` oklch(0.577 0.245 27.325) | Reject button, remove step button — destructive actions only |
| Muted | `--muted-foreground` oklch(0.556 0 0) | PROPOSED step name text, "already covered" items, dashed icon stroke |
| Alert overlay | `bg-amber-50` | Step row background when step has active alert — unchanged from Phase 5 |

Accent (`--primary`) is reserved for:
- The "Confirm" button on each PROPOSED step row (sole primary CTA per row)
- The "Save Step" button in InlineStepEdit when editing a PROPOSED step before confirming

Destructive (`--destructive`) is reserved for:
- The "Reject" button/icon on each PROPOSED step row
- The "Remove Step" button in confirmation dialogs

No new semantic colors are introduced. The existing `--severity-*` tokens (overdue/missing/out-of-order/completed) are not used in Phase 6 UI elements.

Source: `src/app.css` CSS custom properties, `StepRow.tsx` `bg-amber-50` alert pattern, `PathwayEditor.tsx` `variant="destructive"` pattern.

## Interaction Contracts

### 1. PROPOSED Step Row (AI-extracted)

**Visual state:** Dashed circle icon (`icon-dashed` class on `Circle` from lucide) + muted step name text + "AI Proposed" `Badge variant="outline"` + source document link as `text-xs text-muted-foreground` below step name.

**Structure within StepRow:**
```
[dashed circle icon] [step name — muted] [AI Proposed badge]
                     [Source: {document filename} — text-xs muted]
                     [timing info — text-xs muted]
[Confirm button] [Edit button] [Reject button]
```

**Confirm button:** `Button variant="default" size="sm" className="h-8 px-3"` with `Check` lucide icon + label "Confirm". Clicking transitions step to ACTIVE status immediately (optimistic update via TanStack Query invalidation).

**Edit button:** `Button variant="ghost" size="sm" className="h-8 w-8 p-0"` with `Pencil` lucide icon. Opens existing `InlineStepEdit` form inline — reuses Phase 5 edit form entirely. "Save Step" button in InlineStepEdit saves the edits but does NOT confirm the step; step remains PROPOSED until Confirm is pressed.

**Reject button:** `Button variant="ghost" size="sm" className="h-8 w-8 p-0 text-destructive hover:text-destructive"` with `X` lucide icon. Clicking opens a confirmation dialog before executing rejection (see Reject Confirmation Dialog below).

**Proposed step row background:** No background tint — inherits default `--background`. The dashed icon + muted text + "AI Proposed" badge provide sufficient visual distinction without a colored row background.

**Proposed step row border:** `rounded-md border border-dashed` — adds dashed border to the entire step row `<li>` element. Only PROPOSED status rows use `border-dashed`; all other statuses use no border (existing behavior).

### 2. Confirm interaction

**Trigger:** Nurse clicks "Confirm" button on a PROPOSED step row.

**Optimistic behavior:** Step row immediately transitions to ACTIVE visual state (solid circle icon, semibold text, no AI Proposed badge, no confirm/edit/reject buttons, standard edit/skip/remove buttons appear). TanStack Query invalidates the pathway query on settlement.

**On error:** Toast notification: "Could not confirm step. Please try again." Step reverts to PROPOSED visual state. No dialog required — non-destructive action.

**Edges:** Proposed edges to/from this step are confirmed automatically by the backend on step confirmation. No additional nurse action needed. The pathway DAG view re-renders with updated topological order after query invalidation.

### 3. Reject Confirmation Dialog

**Trigger:** Nurse clicks Reject button on a PROPOSED step row.

**Dialog:** `Dialog` component, `max-w-md`. No free-text reason field — rejection is one-click after confirmation. This keeps the UX lightweight.

Content:
- Title: `Remove proposal for "{step name}"?`
- Description: `This step will be marked as rejected. It won't appear in the pathway, and the AI won't re-propose it from future document uploads.`
- Footer buttons: `Button variant="outline"` ("Keep Proposal") and `Button variant="destructive"` ("Reject Proposal")

**On confirm:** Step transitions to REJECTED status (hidden from view). TanStack Query invalidates pathway query. No toast needed — the step disappearing from the list is sufficient feedback.

**On error:** Dialog remains open. Error message appears above footer: `text-destructive text-sm` — "An error occurred. Your changes were not saved. Please try again." Matches existing PathwayEditor remove dialog error pattern.

### 4. Edit Before Confirm (PROPOSED step edit)

PROPOSED steps in the pathway list show the "Edit" (Pencil) button alongside Confirm and Reject at all times — no separate edit mode toggle required. Clicking Edit expands the step into `InlineStepEdit` form inline (same component used for ACTIVE steps in PathwayEditor).

The step name, event type, and window days fields are pre-populated from Claude's extraction output. Fields Claude could not determine render empty (same as blank manual entry).

"Save Step" saves edits, returns to PROPOSED row view with updated values. "Discard" restores original extracted values.

### 5. "Already Covered" Informational Section

Rendered below the pathway step list, above the "Add Step" button, only when the last document upload produced at least one duplicate detection.

**Visual design:**
- `Card` with `CardHeader` ("Steps already in pathway") and `CardContent`
- Header: `text-xs font-semibold text-muted-foreground uppercase tracking-wide`
- Body: flat list of step names, each as `text-sm text-muted-foreground`, separated by commas or stacked as `space-y-1`
- Collapsible — initially collapsed. `CollapsibleTrigger` shows "N steps already tracked" summary, expands to show full list.
- No action buttons — informational only.

**Placement:** Below the step list `<ol>`, above the "Add Step" button. Rendered only when `alreadyCoveredSteps.length > 0`.

### 6. "Show Rejected Steps" Toggle

Below the "already covered" section (or below "Add Step" if no already-covered). Uses `Collapsible` with a ghost button trigger: `Button variant="ghost" size="sm"` showing "Show {N} rejected step{s}" when collapsed, "Hide rejected steps" when expanded.

Rejected steps render as a flat list inside the collapsible — same visual treatment as PROPOSED but with a strikethrough on the step name and no action buttons. `Badge variant="outline"` showing "Rejected" instead of "AI Proposed".

**Default state:** Collapsed (hidden). Only shown if `rejectedSteps.length > 0`.

### 7. Extraction In Progress (loading state)

During document upload → classification → extraction pipeline, the pathway panel shows:
- Existing skeleton loading state (5 × `Skeleton h-[44px]`) while awaiting the initial pathway load
- After pathway loads and extraction is still in-progress: `Progress` component below the step list showing indeterminate state with label `text-xs text-muted-foreground "Extracting pathway steps from document..."`
- On extraction completion: `Progress` component removes; new PROPOSED steps appear in the step list via query invalidation.

If extraction fails silently (feature flag off, circuit breaker open, Claude error): no loading indicator shown. Steps do not change. No user-visible error — matches the existing silent-fallback pattern from Phase 4 document classification.

### 8. Proposed Edges (visual rendering)

Proposed edges (edges connecting PROPOSED steps to existing steps, or between PROPOSED steps) use the same `EdgeEditor` component from Phase 5. No new edge rendering is added in Phase 6 — edges are bundled with step confirmation and are not individually displayed as proposed.

The `EdgeEditor` will show confirmed edges after step confirmation triggers query invalidation. No intermediate "proposed edge" visual state is required at this phase.

## Accessibility Contract

- All confirm/reject/edit icon buttons have `aria-label` including the step name: `aria-label="Confirm {step name}"`, `aria-label="Reject proposal for {step name}"`, `aria-label="Edit {step name}"`.
- PROPOSED step rows retain `aria-level={step.depth + 1}` from existing StepRow.
- Reject confirmation dialog uses `Dialog` (Radix Dialog) — focus traps automatically, Escape key closes, focus returns to trigger on close.
- "Already covered" collapsible uses `Collapsible` (Radix Collapsible) — keyboard accessible.
- `aria-live="polite"` on the pathway list container (`PathwayDAGView`) — already present, catches step list changes after confirm/reject.
- Progress indicator for extraction uses `role="status"` and `aria-label="Extracting pathway steps"`.
