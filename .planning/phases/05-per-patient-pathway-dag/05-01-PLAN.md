- `created_by UUID NOT NULL`

Create `patient_pathway_steps` table:
- `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
- `pathway_id UUID NOT NULL REFERENCES patient_pathways(id) ON DELETE CASCADE`
- `name VARCHAR(255) NOT NULL`
- `description TEXT`
- `event_type care_event_type` (uses existing PostgreSQL enum)
- `window_days INTEGER`
- `required BOOLEAN NOT NULL DEFAULT true`
- `status pathway_step_status NOT NULL DEFAULT 'ACTIVE'`
- `skip_reason TEXT`
- `alert_text TEXT`
- `suggested_action TEXT`

- `completed_at TIMESTAMP WITH TIME ZONE`
- `completed_care_event_id UUID REFERENCES care_events(id)` (explicit linkage per RESEARCH open question 1)
- `created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()`
- `created_by UUID NOT NULL`
- `version INTEGER NOT NULL DEFAULT 0` (optimistic locking)

Create `patient_pathway_edges` table:
- `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`

**V15__migrate_patients_to_per_patient_pathways.sql:**

This migration converts ALL existing patients (per D-08). Logic:

1. For each patient in `patients` table (regardless of status):
   - Find the matching `pathway_templates` row by `cancer_type`
   - If template exists: INSERT into `patient_pathways` with `source_template_id`, `source_template_version = 1`
   - Use `jsonb_array_elements(template_data)` to iterate template steps
   - For each template step: INSERT into `patient_pathway_steps` with fields mapped from JSONB (name, description, event_type cast to care_event_type, window_days, required, alert_text, suggested_action, source_template_step_id = stepId)
   - For each step with non-empty `prerequisites` array: use `jsonb_array_elements_text(step->'prerequisites')` and INSERT edges into `patient_pathway_edges` where source_step is the prerequisite's per-patient step and target_step is the current step
   - `created_by` = `'00000000-0000-0000-0000-000000000000'::uuid` (system migration actor)

2. Handle physician overrides (D-04):
   - For each row in `physician_overrides`: find the per-patient step matching `(patient_id, pathway_step_id = source_template_step_id)` and UPDATE its status to 'SKIPPED', set skip_reason to the override's `reason` field

3. Handle completed care events:
   - For each patient's care events with status 'COMPLETED': find per-patient steps matching by event_type (in step number order for disambiguation), mark as COMPLETED with `completed_at = care_event.event_date::timestamptz`, `completed_care_event_id = care_event.id`
   - Use a window function to handle multiple events of the same type: assign in step order

Use CTEs and temp tables for the step ID mapping. The entire migration is a single transaction (Flyway default).

Handle edge cases:
- Patient with no matching template (cancer_type not in pathway_templates): create patient_pathways row with NULL source_template_id, 0 steps
- Template with empty/null template_data: create pathway with 0 steps (use COALESCE and check array length)
- INACTIVE patients: still migrate (audit trail retention per D-08 context)
