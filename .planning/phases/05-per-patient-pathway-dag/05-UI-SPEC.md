## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, branching indicator padding |
| sm | 8px | Compact element spacing, step list item gaps |
| md | 16px | Default element spacing, card content padding |
| lg | 24px | Section padding, step-to-step vertical spacing in editor mode |
| xl | 32px | Layout gaps between split columns |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- DAG depth indentation: 24px per depth level (1 depth = `pl-6` / 24px left padding). Maximum 4 levels deep for tablet readability at 768px minimum width.
- Touch targets: all interactive elements (edit icons, remove buttons, add step button) minimum 44x44px hit area, consistent with existing Phase 3 pattern (`min-h-[44px]` on step list items).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label / Step Name | 14px | 600 (semibold) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 28px | 600 (semibold) | 1.2 |

Weights declared: **2** -- 400 (regular) and 600 (semibold). No other weights are permitted.

Source: Existing codebase uses `text-sm` (14px) for step names and event details, `text-xl font-semibold` (20px/600) for card headings, `text-3xl font-semibold tracking-tight` (28px-ish/600) for page titles. This contract codifies the established pattern and collapses the previous `font-medium` (500) usage into `font-semibold` (600) for consistency.

Additional typography rules for Phase 5:
- Step name in view mode: 14px / weight 600 / `leading-snug` (use `font-semibold text-sm leading-snug` -- replaces previous `font-medium`)
- Step name in edit mode: 14px / weight 400 / standard line-height (editable input inherits body style)
- Branching indicators (`---`, `---`): 14px / weight 400 / `font-mono` / `text-muted-foreground`
- Timing info below step name: 12px (`text-xs`) / weight 400 / `text-muted-foreground`
- SKIPPED step name: 14px / weight 400 / `line-through text-muted-foreground`
- PROPOSED step name: 14px / weight 400 / `text-muted-foreground` (no line-through, visually lighter than ACTIVE)

### Phase 5 Step Status Colors

These extend the existing clinical severity palette in `App.css`:

| Status | Icon | Color | Tailwind Class | CSS Variable |
|--------|------|-------|----------------|--------------|
| COMPLETED | CheckCircle2 (lucide) | Green | `text-green-600` | -- |
| ACTIVE (no alert) | Circle (lucide) | Blue | `text-blue-500` | -- |
| ACTIVE (overdue/alert) | AlertTriangle (lucide) | Red | `text-red-500` | -- |
| PROPOSED | Circle (lucide, dashed via `stroke-dasharray`) | Gray | `text-muted-foreground` | -- |
| SKIPPED | MinusCircle (lucide) | Gray | `text-muted-foreground` | -- |

Notes:
- COMPLETED green (`text-green-600`) matches existing `CheckCircle2` usage in `$patientId.tsx` line 70.
- ACTIVE blue (`text-blue-500`) replaces the generic `Circle` gray used for UPCOMING in Phase 3. This is the only color change from Phase 3.
- ACTIVE with alert keeps the existing `AlertTriangle text-red-500` pattern (line 66 of `$patientId.tsx`).
- PROPOSED uses a dashed outline effect. Implementation: wrap the Circle icon's SVG with `stroke-dasharray="4 2"` via a custom CSS class `.icon-dashed { stroke-dasharray: 4 2; }` applied alongside `text-muted-foreground`.
- SKIPPED uses `MinusCircle` (lucide) in `text-muted-foreground` with the step name rendered in `line-through`.
- Step rows with active alerts keep the existing `bg-amber-50` highlight (line 330 of `$patientId.tsx`).

### DAG Depth Visual Encoding

Parallel steps at the same depth level are grouped visually with branching indicators:

```
1. Initial Consultation          (depth 0, no indent)
  +-- 2. CT Scan                 (depth 1, 24px indent)
  +-- 3. Blood Work              (depth 1, 24px indent, parallel with CT Scan)
      +-- 4. Chemotherapy         (depth 2, 48px indent, depends on both)
```

- Branching connector characters: Unicode box-drawing `\u251C\u2500\u2500` (middle child), `\u2514\u2500\u2500` (last child at same depth).
- Connector color: `text-muted-foreground` (oklch 0.556 0 0).
- No horizontal lines or SVG connectors --- purely indentation + text characters, matching D-13 decision for tablet friendliness.

### Focal Point

**Primary focal point (patient detail screen):** PathwayDAGView step list with status icons in the left column. The status icon column (leftmost element per step row) is the primary scanning axis -- nurses scan vertically down the icon column to identify which steps need attention. Green checkmarks are "done," blue circles are "in progress," red triangles are "needs action."

## Interaction Contracts

### 1. Template Picker (Patient Creation Wizard)

**Location:** Step 2 of `PatientWizard`, below Cancer Type selector.

**Behavior:**
- Renders after cancer type is selected (needs cancer type to show template name).
- Two radio options: "Start from [Cancer Type] template" (pre-selected, default) and "Build from documents (empty pathway)".
- Radio group uses shadcn RadioGroup or simple styled radio inputs.
- Selection stored as `pathwayMode: 'template' | 'empty'` in Step 2 form data.
- When "Build from documents" is selected, show a muted helper text: "Steps will be added manually or extracted from uploaded documents."
- On submit, `pathwayMode` is sent to the create patient endpoint.

**States:**
- No cancer type selected: template picker hidden.
- Cancer type selected: template picker visible, template option pre-selected.
- Template option: shows template name (e.g., "Breast Cancer Pathway -- 12 steps").
- Empty option: shows "Build from documents (empty pathway)".

### 2. Enhanced Vertical List (PathwayDAGView)

**Location:** Left column of patient detail split layout, inside Pathway Status card.

**Behavior:**
- Steps sorted by topological order (server provides `depth` and `sortOrder` per step).
- Each step indented by `depth * 24px` from the left edge.
- Branching indicators rendered as monospace text prefix: `\u251C\u2500\u2500` for non-last children, `\u2514\u2500\u2500` for last child at depth.
- Each step row shows: status icon, step name, timing info, active alert badge (if applicable).
- PROPOSED steps rendered with dashed icon and `text-muted-foreground` name. Tooltip on hover explains "Pending confirmation".
- SKIPPED steps rendered with `MinusCircle` icon, `line-through` name, and "Skipped" outline badge with reason on hover tooltip.
- Clicking a step row does nothing in view mode (no navigation, no expansion).

**States:**
- Loading: 5 skeleton rows (matches existing pattern).
- Empty (no steps): empty state message with prompt to use Edit Pathway.
- Populated: tiered step list with status indicators.
- Step with active alert: `bg-amber-50` row highlight (existing pattern).

### 3. Inline Pathway Editor (PathwayEditor)

**Location:** Replaces `PathwayDAGView` in left column when "Edit Pathway" is toggled.

**Toggle behavior:**
- "Edit Pathway" button in card header, `variant="outline"`, icon `Pencil` (lucide).
- When toggled ON: button changes to "Done Editing" with `variant="default"` (primary), icon `Check` (lucide). Pathway view becomes editor.
- When toggled OFF: editor unmounts, view re-renders with fresh data from server.
- PROPOSED steps are visible but not editable in Phase 5 (Phase 6 adds confirm/reject).

**Step editing:**
- Each step row in editor mode gets two icon buttons on the right: `Pencil` (edit) and `Trash2` (remove).
- Edit icon: step row transforms into inline edit fields (step name Input, event type Select, time window Input). "Save Step" and "Discard" text buttons below the fields.
- Remove icon: opens `Dialog` confirmation with destructive copy.
- "Add Step" button at the bottom of the step list: expands to show `AddStepForm` inline (not a dialog). Form has: Step Name (required Input), Expected Event Type (Select from known types), Time Window in days (number Input, optional). "Add Step" and "Discard" buttons.
- New steps are added as root steps (depth 0) by default. The nurse can add edges afterward via EdgeEditor.

**Edge editing (EdgeEditor):**
- Rendered below the step list in editor mode.
- Collapsible section with header "Dependencies" and a toggle chevron.
- Shows existing edges as a list: "[Source Step] must complete before [Target Step]" with a `X` remove button.
- "Add Dependency" button: shows two Select dropdowns ("After step:" and "Before step:") and an "Add Dependency" button.
- Cycle detection: if the new edge would create a cycle, show inline error message with the cycle detection copy from Copywriting Contract. The "Add Dependency" button is disabled.
- Edge removal: immediate, no confirmation dialog (reversible by re-adding).

**Optimistic updates:**
- Step add/edit/remove and edge add/remove use TanStack Query `useMutation` with `onSuccess` invalidation of `['patients', patientId, 'pathway-status']` and `['patients', patientId, 'pathway-steps']`.
- No optimistic UI for adds (wait for server confirmation to get step ID).
- Optimistic removal: step disappears immediately, rolls back on error with toast/inline error.

### 4. Step Status Transitions

**Available actions per status (in editor mode):**

| Current Status | Available Actions |
|----------------|-------------------|
| ACTIVE | Edit, Remove, Skip |
| COMPLETED | Remove (with warning: "This step has a recorded care event.") |
| PROPOSED | View only (Phase 5), Remove |
| SKIPPED | Unskip (restore to ACTIVE), Remove |

- "Skip" action: replaces the old physician override mechanism. Opens `SkipStepDialog` requiring a reason. On confirm, step status becomes SKIPPED.
- "Unskip" action: text button on SKIPPED steps in editor mode. Restores to ACTIVE. No confirmation needed.

### 5. Split Layout Behavior

**Desktop (1024px+):**
- Grid: `grid-cols-5` (existing). Pathway view/editor: `col-span-3`. Care events: `col-span-2`.
- Gap: `gap-6` (24px, existing).

**Tablet (768px -- 1023px):**
- Stacks vertically: single column, pathway on top, care events below.
- Maximum DAG depth indentation: 4 levels * 24px = 96px. At 768px width minus card padding (32px), the remaining content area is ~640px, which accommodates 96px indent + step content comfortably.

**Below 768px:**
- Same vertical stack. Branching indicators still render. Indentation capped at 4 levels.

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| Step status announced | `aria-label` on status icons: "Completed", "Active", "Active - overdue", "Proposed - pending confirmation", "Skipped" |
| Edit mode announced | `aria-live="polite"` region around pathway card. Announce "Pathway editor opened" / "Pathway editor closed" on toggle |
| Inline edit focus | Focus moves to step name Input when edit icon is clicked. Focus returns to edit icon when canceled |
| Remove confirmation | Dialog traps focus. Escape dismisses. Focus returns to the list after removal |
| Keyboard navigation | Tab through step rows, edit/remove actions reachable via Tab. Enter/Space activates buttons |
| Branching indicators | Decorative only, use `aria-hidden="true"`. Step depth conveyed via `aria-level` attribute on step row |
| PROPOSED tooltip | Tooltip trigger via `Tooltip` component (keyboard accessible via focus) |
| Color not sole indicator | Status communicated via icon shape (CheckCircle2 vs Circle vs MinusCircle vs AlertTriangle) AND text (timing info, "Skipped" badge), not color alone |
| Edit icon button label | `aria-label="Edit [Step Name]"` on the Pencil icon button in each step row (editor mode). Dynamic -- interpolates the step name so screen readers distinguish between rows |
| Remove icon button label | `aria-label="Remove [Step Name]"` on the Trash2 icon button in each step row (editor mode). Dynamic -- interpolates the step name so screen readers distinguish between rows |
